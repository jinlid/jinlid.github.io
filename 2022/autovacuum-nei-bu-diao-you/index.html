autovacuum内部调优

postgresql
postgresql
发布日期:   2022-01-12

PostgreSQL提供了一大堆系统视图，它们让管理员和开发者之类的人可以深入地审视系统中究竟在做什么。可以了解各种维度的统计信息的解读（table，index，database，replication，sql），锁解读信息，磁盘使用，数据库级别或者事务级别或者会话级别的活动信息统计，历史数据库活动统计报告。

PostgreSQL数据库的性能可能会因为死元组而受到影响，因为它们会继续占用空间并导致膨胀。我们在以前的博客文章中介绍了VACUUM和bloat。现在，是时候看看用于postgres的autovacuum，以及维护高要求应用程序所需的高性能PostgreSQL数据库的内部知识。

什么是autovacuum?

autovacuum是一个后台实用程序进程，当你启动PostgreSQL时自动启动。正如您在下面的日志中看到的，pid为2862的postmaster(父PostgreSQL进程)已经启动了pid为2868的autovacuum launcher进程。要启动autovacuum，必须将autovacuum参数设置为ON。事实上，在生产系统中，您不应该将其设置为OFF，除非您100%确定您正在做什么及其含义。

avi@percona:~$ps -eaf | egrep "/post|autovacuum"
postgres  2862     1  0 Jun17 pts/0    00:00:11 /usr/pgsql-10/bin/postgres -D /var/lib/pgsql/10/data
postgres  2868  2862  0 Jun17 ?        00:00:10 postgres: autovacuum launcher process   
postgres 15427  4398  0 18:35 pts/1    00:00:00 grep -E --color=auto /post|autovacuum

为什么需要autovacuum?

我们需要VACUUM来移除死元组，这样被死元组占用的空间就可以被表重用，用于将来的插入/更新。要了解更多关于死元组和膨胀的信息，请阅读我们[以前的博客文章]https://www.percona.com/blog/2018/08/06/basic-understanding-bloat-vacuum-postgresql-mvcc/)We也需要在更新表统计信息的表上进行`ANALYZE`，这样优化器就可以为SQL语句选择最优的执行计划。postgres中的`autovacuum`负责对表进行真空和分析。

postgres中存在另一个名为“Stats Collector”的后台进程，用于跟踪使用情况和活动信息。autovacuum启动程序使用此过程收集的信息来确定autovacuum的候选表列表。PostgreSQL识别需要自动抽取或分析的表，但仅当启用autovacuum时。这确保了postgres自我修复，并阻止数据库产生更多的膨胀/碎片。

在PostgreSQL中启用autovacuum所需的参数是：

autovacuum = on  # ( ON by default )
track_counts = on # ( ON by default )

track_counts由统计收集器使用。如果不这样做，autovacuum就不能访问候选表。

记录autovacuum

最后，您可能希望记录autovacuum花费更多时间的表。在这种情况下，将参数log_autovacuum_min_duration设置为一个值(默认为毫秒)，这样任何运行时间超过这个值的autovacuum都会被记录到PostgreSQL日志文件中。这可能有助于适当地调整您的表级自动真空设置。

# Setting this parameter to 0 logs every autovacuum to the log file.
log_autovacuum_min_duration = '250ms' # Or 1s, 1min, 1h, 1d

这里是一个例子，日志自动真空真空和分析

pages: 0 removed, 1190 remain, 0 skipped due to pins, 0 skipped frozen
tuples: 110008 removed, 110008 remain, 0 are dead but not yet removable

< 2018-08-06 07:22:35.199 EDT > LOG: automatic analyze of table "vactest.scott.employee" system usage: CPU 0.00s/0.02u sec elapsed 0.15 sec

PostgreSQL什么时候在表上运行autovacuum ?

如前所述，postgres中的autovacuum指的是automatic VACUUM和ANALYZE，而不仅仅是VACUUM。根据下面的数学方程，autovacuum或analyze在表上运行。

有效工作台液位自动真空阈值的计算公式为:

Autovacuum VACUUM thresold for a table = autovacuum_vacuum_scale_factor * number of tuples + autovacuum_vacuum_threshold

根据上面的公式，很明显，如果由于更新和删除，表中死亡元组的实际数量超过了这个有效阈值，那么该表将成为autovacuum真空的候选者。

Autovacuum ANALYZE threshold for a table = autovacuum_analyze_scale_factor * number of tuples + autovacuum_analyze_threshold

上面的公式表明，任何插入/删除/更新总数超过这个阈值的表(自上次分析以来)都有资格进行autovacuum分析。

让我们详细了解这些参数。

  • **autovacuum_vacuum_scale_factor **or autovacuum_analyze_scale_factor : 表的一部分记录将被添加到公式中的内容。例如，值0.2等于表记录的20%。
  • autovacuum_vacuum_threshold or autovacuum_analyze_threshold : 触发自动真空所需的废弃记录或dml的最小数量。

让我们考虑一个表:percona。员工有1000条记录和以下autovacuum参数。

autovacuum_vacuum_scale_factor = 0.2
autovacuum_vacuum_threshold = 50
autovacuum_analyze_scale_factor = 0.1
autovacuum_analyze_threshold = 50

参考上述数学公式，

Table : percona.employee becomes a candidate for autovacuum Vacuum when,
Total number of Obsolete records = (0.2 * 1000) + 50 = 250

Table : percona.employee becomes a candidate for autovacuum ANALYZE when,
Total number of Inserts/Deletes/Updates = (0.1 * 1000) + 50 = 150

在PostgreSQL中调整autovacuum

我们需要明白这些都是全局设定。这些设置适用于实例中的所有数据库。这意味着，无论表的大小如何，如果达到上述公式，表就有资格进行自动vacuum或analyze。

这有问题吗?

考虑一个有10条记录的表和一个有100万条记录的表。尽管具有一百万条记录的表可能更频繁地涉及事务，但对于只有十条记录的表，vacuum或analyze自动运行的频率可能更高。

因此，PostgreSQL允许你配置单独的表级自动真空设置，绕过全局设置。

ALTER TABLE scott.employee SET (autovacuum_vacuum_scale_factor = 0, autovacuum_vacuum_threshold = 100);

Output Log
----------
avi@percona:~$psql -d percona
psql (10.4)
Type "help" for help.

percona=# ALTER TABLE scott.employee SET (autovacuum_vacuum_scale_factor = 0, autovacuum_vacuum_threshold = 100);
ALTER TABLE

上面的设置在表上autovacuum。员工仅一次就有超过100条过时的记录。

如何识别需要调优autovacuum设置的表?

为了单独为表调优autovacuum，必须知道在一定时间间隔内对表进行插入/删除/更新的次数。您还可以查看postgres目录视图：pg_stat_user_tables以获取该信息。

percona=# SELECT n_tup_ins as "inserts",n_tup_upd as "updates",n_tup_del as "deletes", n_live_tup as "live_tuples", n_dead_tup as "dead_tuples"
FROM pg_stat_user_tables
WHERE schemaname = 'scott' and relname = 'employee';
 inserts | updates | deletes | live_tuples | dead_tuples 
---------+---------+---------+-------------+-------------
      30 |      40 |       9 |          21 |          39
(1 row)

正如在上面的日志中观察到的那样，在一定的时间间隔内对该数据进行快照将有助于您了解每个表上dml的频率。反过来，这将帮助您调整各个表的autovacuum设置。

一次可以运行多少个autovacuum进程?

在可能包含多个数据库的实例/集群中，一次运行的autovacuum进程数量不能超过autovacuum_max_workers。autovacuum启动程序后台进程为需要vacuum或analyze的表启动工作进程。如果有四个数据库，autovacuum_max_workers设置为3，那么，第四个数据库必须等待，直到一个现有的工作进程空闲。

在开始下一次autovacuum之前，它会等待autovacuum_naptime，在大多数版本上默认是1分钟。如果您有三个数据库，那么下一次autovacuum将等待60/3秒。因此，启动下一次autovacuum之前的等待时间总是(autovacuum_naptime/N)，其中N是实例中数据库的总数。

增加autovacuum_max_workers是否增加了可以并行运行的autovacuum进程的数量?
不。这将在接下来的几行中得到更好的解释。

VACUUM IO密集吗?

autovacuum可以被认为是一种清理。如前所述，每个表有一个工作进程。autovacuum从磁盘读取8KB(默认block_size)的表页，并修改/写入包含死元组的页。这涉及到读IO和写IO。因此，这可能是一个IO密集型操作，在事务高峰期间，在一个包含许多死元组的巨大表上运行一个autovacuum。为了避免这个问题，我们设置了一些参数来最小化真空对IO的影响。

以下是用于调优autovacuum IO的参数

  • autovacuum_vacuum_cost_limit : autovacuum可以达到的总成本限制(所有autovacuum任务的总和)
  • autovacuum_vacuum_cost_delay : 当清理达到autovacuum_vacuum_cost_limit cost时，autovacuum将休眠这些毫秒。
  • vacuum_cost_page_hit : 读取已经在共享缓冲区中且不需要读取磁盘的页面的开销。
  • vacuum_cost_page_miss : 获取不在共享缓冲区中的页面的开销。
  • vacuum_cost_page_dirty : 在每页中发现死元组时写入该页的开销。
Default Values for the parameters discussed above.
------------------------------------------------------
autovacuum_vacuum_cost_limit = -1 (So, it defaults to vacuum_cost_limit) = 200
autovacuum_vacuum_cost_delay = 20ms
vacuum_cost_page_hit = 1
vacuum_cost_page_miss = 10
vacuum_cost_page_dirty = 20

考虑在工作台percona.employee上运行autovacuum VACUUM。

让我们想象一秒钟内会发生什么。(1秒= 1000毫秒)

在读延迟为0毫秒的最佳情况下，autovacuum可以唤醒并进入睡眠50次(1000毫秒/ 20毫秒)，因为唤醒之间的延迟需要为20毫秒。

1 second = 1000 milliseconds = 50 *&nbsp;autovacuum_vacuum_cost_delay

由于在shared_buffers中每次读取一个页面的代价是1，因此在每次唤醒中可以读取200个页面，在50次唤醒中可以读取50*200个页面。

如果所有带死元组的页面都在共享缓冲区中找到，autovacuum_vacuum_cost_delay为20ms，那么它可以在每轮读取((200 / vacuum_cost_hit) * 8) KB，需要等待autovacuum_vacuum_cost_delay的时间量。

因此，考虑到block_size为8192字节，autovacuum最多可以读取:50 * 200 * 8 KB = 78.13 MB每秒(如果块已经在shared_buffers中找到)。

如果块不在共享缓冲区中，需要从磁盘读取，autovacuum可以读取:50 * ((200 / vacuum_cost_page_miss) * 8) KB = 7.81 MB / s。

我们上面看到的所有信息都是针对读IO的。

现在，为了从页/块中删除死元组，写操作的代价是：vacuum_cost_page_dirty，默认设置为20。

autovacuum最多可以写/脏:50 * ((200 / vacuum_cost_page_dirty) * 8) KB = 3.9 MB/s。

通常，这个成本会平均分配给实例中运行的所有autovacuum_max_workers数量的autovacuum进程。因此，增加autovacuum_max_workers可能会延迟当前正在运行的autovacuum workers的autovacuum执行。增加autovacuum_vacuum_cost_limit可能会导致IO瓶颈。需要注意的一点是，可以通过设置单个表的存储参数来覆盖这种行为，这随后会忽略全局设置。

postgres=# alter table percona.employee set (autovacuum_vacuum_cost_limit = 500);
ALTER TABLE
postgres=# alter table percona.employee set (autovacuum_vacuum_cost_delay = 10);
ALTER TABLE
postgres=# 
postgres=# \d+ percona.employee
Table "percona.employee"
Column | Type | Collation | Nullable | Default | Storage | Stats target | Description 
--------+---------+-----------+----------+---------+---------+--------------+-------------
id | integer | | | | plain | | 
Options: autovacuum_vacuum_threshold=10000, autovacuum_vacuum_cost_limit=500, autovacuum_vacuum_cost_delay=10

因此，在繁忙的OLTP数据库上，总是有一种策略，在低峰值窗口期间，对经常被dml命中的表实现手动VACUUM。你可以有尽可能多的并行vacuum作业，当你手动运行后，设置相关的autovacuum_*设置。出于这个原因，定期手动vacuum作业总是建议与微调autovacuum设置。
