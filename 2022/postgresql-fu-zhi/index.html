1 理解复制概念

复制分为同步复制和异步复制

这是什么意思？假设我们有两台服务器，我们想将数据从一台服务器（主服务器）复制到第二台服务器（从服务器）。下图说明了同步和异步复制的概念：

在异步复制的情况下，可以在主服务器上提交事务后复制数据。换句话说，从库永远不会领先于主库，而在写作的情况下，通常会落后于主库一点，这种延迟称为滞后。同步复制强制执行更高的一致性规则。如果您决定同步复制，系统必须确保事务写入的数据在事务提交时至少在两个服务器上。这意味着从服务器不会落后于主服务器，最终用户看到的数据在两个服务器上都是相同的。一些系统也会使用一个仲裁服务器来决定。因此，它并不总是只涉及两个或更多服务器。如果使用仲裁，则必须有超过一半的服务器同意集群内的操作。

了解复制和数据丢失

当事务从主服务器复制到从服务器时，必须考虑很多事情，尤其是涉及数据丢失等事情时。

让我们假设我们正在以下列方式异步复制数据：

  1. 一个事务被发送给master。
  2. 它提交给 master。
  3. master 在提交发送给 slave 之前就死了。
  4. slave永远不会得到这个事务。

在异步复制的情况下，存在一个窗口（滞后），在此期间数据基本上可能会丢失。此窗口的大小可能因设置类型而异。它的大小可以很短（可能短至几毫秒）或很长（几分钟、几小时或几天）。重要的事实是数据可能会丢失。小的滞后只会降低数据丢失的可能性，但是，任何大于零的滞后都容易导致数据丢失。

如果要确保数据永远不会丢失，则必须切换到同步复制。正如您在本章中已经看到的，同步事务之所以是同步的，是因为它只有在至少提交到两台服务器。

考虑性能问题

通过网络发送不必要的消息可能既昂贵又耗时。如果以同步方式复制事务，PostgreSQL 必须确保数据已到达第二个节点，这将导致延迟问题。同步复制在很多方面都比异步复制更昂贵，因此如果确实需要并证明这种开销是合理的，人们应该三思而后行。因此只在真正需要时才使用同步复制。

单主复制与多主复制

对各种复制设置进行分类的第二种方法是区分单主复制和多主复制。单主机意味着写入可以只发送到一台服务器，该服务器将数据分发到设置内的从机。从机只能接收读取而不能接收写入。

与单主复制相比，多主复制允许写入集群内的所有服务器。下图显示了事物在概念层面上的工作方式：

能够写入集群内的任何节点听起来像是一种优势，但不一定是。这样做的原因是多主复制给系统增加了很多复杂性。在只有一个master的情况下，哪些数据是正确的，数据会往哪个方向流动，一目了然，复制时很少会发生冲突。

多主复制完全不同，因为写入可以同时到多个节点，集群必须完全意识到冲突并优雅地处理它们。一种替代方法是使用锁来解决问题，但这种方法有其自身的问题。请记住，解决冲突的需要会导致网络流量增加，这会立即变成由延迟引起的可伸缩性问题。

逻辑与物理复制

对复制进行分类的另一种方法是区分逻辑复制和物理复制。区别很细微但非常重要：物理复制意味着系统会将数据按原样移动到远程机器。因此，如果插入了某些内容，远程框将获取二进制格式的数据，而不是通过 SQL。逻辑复制就是复制一个变化，相当于数据进来了。

让我们看一个例子来充分理解其中的区别：

test=# CREATE TABLE t_test (t date);
CREATE TABLE
test=# INSERT INTO t_test VALUES (now()) 
RETURNING *;
t
------------
 2013-02-08
(1 row)
INSERT 0 1

我们看到这里发生了两个事务：第一个事务创建了一个表。完成后，第二个事务将一个简单的⽇期添加到表中并提交。在逻辑复制的情况下，更改将以逻辑形式发送到某种队列，因此系统不会发送普通 SQL，但可能发送如下内容：

test=# INSERT INTO t_test VALUES ('2013-02-08');
INSERT 0 1

请注意，函数调用已被替换为实际值。如果从机再次计算now()将是一场彻底的灾难，因为远程盒子上的⽇期可能是一个完全不同的⽇期。一些系统确实使用基于语句的复制作为核心技术。例如，MySQL 使用所谓的 bin‑log 进行复制，这实际上不是二进制，而是某种形式的逻辑复制。

物理复制将以完全不同的方式工作：系统不会发送一些 SQL（或其他任何东西），这在逻辑上是等效的，系统将发送 PostgreSQL 内部所做的二进制更改。以下是我们的两个事务可能触发的一些二进制更改（但到目前为止，还不是完整列表）：

  1. 将一个 8k 的块添加到pg_class并在那里放置一条新记录（以指示该表存在）。
  2. 将行添加到pg_attribute以存储列名。
  3. 在这些表的索引内部执行各种更改。
  4. 记录提交状态等。

物理复制的目标是创建一个在物理层面（大部分）相同的系统副本。这意味着相同的数据将位于所有框的表格内的相同位置。在逻辑复制的情况下，内容应该是相同的，但是否在同一个地方没有区别。何时使用物理复制物理复制使用起来非常方便，而且特别容易设置。当目标是拥有相同的系统副本（进行备份或简单地向上扩展）时，它被广泛使用。

在许多设置中，物理复制是标准方法，它使最终用户面临尽可能低的复杂性。横向扩展数据是理想的。何时使用逻辑复制逻辑复制通常设置起来有点困难，但它提供了更大的灵活性。在升级现有数据库时，这一点也尤为重要。物理复制完全不适合版本跳跃，因为你不能简单地依赖于每个版本的 PostgreSQL 都具有相同的磁盘布局这一事实。

存储格式可能会随着时间而改变，因此二进制副本显然无法从一个版本跳转到下一个版本。逻辑复制允许将数据存储方式与其传输和复制方式分离。通过使用不绑定到特定版本的 PostgreSQL 的中性协议，可以很容易地从一个版本跳到下一个版本。

2 了解PostgreSQL 事务⽇志

在上一章中，我们讨论了各种复制概念。它更像是一个理论概述，以提高您对即将发生的事情的感觉，并且应该向您介绍这个主题。在本章中，我们将更接近实际的解决方案，并了解 PostgreSQL 的内部工作原理以及它对复制的意义。我们将看到所谓的事务⽇志(XLOG)的作用及其运作方式。 XLOG 是 PostgreSQL 复制机制的支柱。了解这部分的工作原理非常重要。

PostgreSQL 磁盘布局

在本章中，我们首先要了解的内容之一是 PostgreSQL 磁盘布局。了解磁盘布局在检查现有设置时非常有帮助，在设计高效、高性能的系统时也很有帮助。

与 Oracle 等其他数据库系统相比，PostgreSQL 将始终依赖文件系统来存储数据。 PostgreSQL 不使用原始设备。这背后的理念是，如果文件系统开发人员已经做好了他的工作，就没有必要一遍又一遍地重新实现文件系统功能。

如果数据目录包含正确的版本号，该文件将在启动时告诉系统。请注意，此文件中只有主要发布版本。很容易在同一主要版本的不同次要版本之间进行复制。

该文件是纯文本可读的。

base – 实际的数据目录

base目录是我们数据目录中最重要的内容之一。它实际上包含真实数据（即表、索引等）。在基本目录中，每个数据库都有自己的子目

不断增长的数据文件

表有时可能非常大，因此或多或少不可能将与表相关的所有数据放入单个数据文件中。为了解决这个问题，PostgreSQL 每添加 1 GB 数据就会添加额外的文件。因此，如果名为16385的文件增长超过 1 GB，则会有一个名为16385.1 的文件。填写完毕后，您将看到一个名为16385.2 的文件，依此类推。这样，PostgreSQL 中的表就可以可靠、安全地扩展，而不必担心某些罕见操作系统或嵌入式设备上的底层文件系统限制。

以块为单位执行 I/O

为了提高 I/O 性能，PostgreSQL 将始终以 8k 块为单位执行 I/O。因此，您会看到您的数据文件总是以 8k 的步长增长。在谈论物理复制时，您必须确保双方（主从）都使用相同的块大小进行编译。除非您已经使用不同的块大小自行显式编译 PostgreSQL，否则您始终可以相信块大小将完全相同并且正好是 8k。

关系分支

除了上一段中讨论的那些数据文件外，PostgreSQL 还将使用相同的编号创建其他文件。截至目前，这些文件用于存储有关表内可用空间的信息（可用空间映射），即所谓的可见性映射等。将来，可能会添加更多类型的关系分支。

global – 全局数据

global将包含全局系统表。这个目录很小，所以你不应该期望过多的存储消耗。

处理独立数据文件

有一件事经常被用户遗忘：单个 PostgreSQL 数据文件基本上或多或少没有价值。如果只有一个数据文件，几乎不可能可靠地恢复数据；尝试从单个数据文件中提取数据很容易以无望的猜测告终。因此，为了读取数据，您需要一个或多或少完整的实例。

pg_clog – 提交⽇志

提交⽇志是工作数据库实例的重要组成部分。它存储该系统上交易的状态。事务可以处于四种状态（TRANSACTION_STATUS_IN_PROGRESS、 TRANSACTION_STATUS_COMMITTED、 TRANSACTION_STATUS_ABORTED和TRANSACTION_STATUS_SUB_COMMITTED），如果事务的提交⽇志状态不可用，PostgreSQL 将不知道是否应该看到一行。

如果数据库实例的提交⽇志由于某种原因被破坏（可能是因为文件系统损坏），接下来的几个小时你会很开心。

如果提交⽇志损坏，我们建议对数据库实例（文件系统）进行快照并伪造提交⽇志；它有时可以帮助从相关数据库实例中检索合理数量的数据。

pg_hba.conf – 基于主机的网络配置

pg_hba.conf文件配置是 PostgreSQL 内部防火墙，代表 PostgreSQL 集群中两个最重要的配置文件之一。它允许用户根据请求的来源定义各种类型的身份验证。对于数据库管理员来说，理解pg_hba.conf文件是至关重要的，因为这个文件决定了一个 slave 是否被允许连接到 master。如果您碰巧在这里遗漏了什么，您可能会在从属⽇志中看到错误消息（例如： no pg_hba.conf entry for …）。

pg_ident.conf – 身份验证

pg_ident.conf文件可以与pg_hba.conf文件结合使用来配置身份验证。

pg_multixact – 多事务状态数据

多事务⽇志管理器在这里有效地处理共享行锁。此目录没有与复制相关的实际影响。

pg_notify – LISTEN/NOTIFY 数据

在此目录中，系统存储有关 LISTEN/NOTIFY（异步后端接口）的信息。没有与复制相关的实际影响。

pg_serial – 有关已提交的可序列化事务的信息

有关可序列化事务的信息存储在这里。我们必须将有关可序列化事务的提交信息存储在磁盘上，以确保长时间运行的事务不会使内存膨胀。内部使用一个简单的 SLRU 结构来跟踪这些事务。

pg_snapshot – 导出的快照

这是一个包含 PostgreSQL 快照管理器所需信息的文件。在某些情况下，必须将快照导出到磁盘以避免进入内存。崩溃后，将自动清除那些导出的快照。

pg_stat_tmp – 临时统计数据

临时统计数据存储在这个文件中。大多数pg_stat_*系统视图都需要此信息（因此也需要提供原始数据的底层函数）。

pg_subtrans – 子事务数据

在这个目录中，我们存储有关子事务的信息。 pg_subtrans （和pg_clog）目录是事务相关信息的永久（磁盘上）存储。每个保存在内存中的页面数量有限，因此在许多情况下不需要实际从磁盘读取。但是，如果有一个长时间运行的事务或后端因打开的事务而闲置，则可能需要能够从磁盘读取和写入此信息。它们还允许信息在服务器重新启动后保持不变。

pg_tblspc – 表空间的符号链接

pg_tblspc目录是一个非常重要的目录。在 PostgreSQL 中，表空间只是一个替代存储位置，由一个目录表示数据。这里重要的是：如果一个数据库实例被完全复制，我们就不能简单地相信集群中的所有服务器都使用完全相同的磁盘布局和相同的存储硬件这一事实。很容易出现主机比从机需要更多 I/O 功率的情况，而从机可能只是充当备份或备用。为了允许用户处理不同的磁盘布局，PostgreSQL 会将符号链接放入pg_tblspc目录。数据库将盲目地遵循这些符号链接来查找这些表空间，而不管它们在哪里。这为最终用户提供了巨大的能力和灵活性。控制存储对于复制和一般性能都是必不可少的。请记住，这些符号链接只能事后更改。应该仔细考虑一下。我们建议仅在真正需要时才使用本节中概述的技巧。对于大多数设置，绝对建议在主服务器和从服务器上使用相同的文件系统布局。这可以大大降低复杂性。

pg_twophase – 有关准备好的语句的信息

PostgreSQL 必须存储有关两阶段提交的信息。虽然两阶段提交可能是一个重要的特性，但目录本身对普通系统管理员来说并不重要。

pg_XLOG – PostgreSQL 事务⽇志 (WAL)

PostgreSQL 事务⽇志是我们在本章中必须讨论的基本目录。 pg_XLOG包含与所谓的 XLOG 相关的所有文件。如果您过去已经使用过 PostgreSQL，您可能熟悉术语WAL （预写⽇志）。 XLOG 和 WAL 是同一事物的两个名称。这同样适用于术语事务⽇志。所有这三个术语都被广泛使用，重要的是要知道它们实际上意味着同一件事。

postgresql.conf – 核心 PostgreSQL 配置文件

最后，还有主 PostgreSQL 配置文件。所有配置参数都可以在postgresql.conf中更改，我们将广泛使用这个文件来设置复制和调整我们的数据库实例，以确保我们的复制设置为我们提供卓越的性能。如果您碰巧使用预构建的二进制文件，您可能找不到 postgresql。 conf直接在你的数据目录中。它更有可能位于 /etc/（在 Linux/Unix 上）的某个子目录中，或者在 Windows 中您选择的位置。精确位置在很大程度上取决于您使用的操作系统类型。数据目录的典型位置是 /var/lib/pgsql/data。但是 postgresql.conf 通常位于 /etc/postgresql/9.X/main/postgresql.conf 下（如在 Ubuntu 和类似系统中）或直接位于 /etc 下。

写入一行数据

一个简单的 INSERT 语句

让我们假设我们正在执行一个简单的INSERT语句，如下所示：

INSERT INTO foo VALUES ('abcd');

可以想象， INSERT操作的目标是以某种方式向现有表中添加一行。我们在上一节关于 PostgreSQL 的磁盘布局中已经看到，每个表都会与磁盘上的一个文件相关联。让我们进行一个心理实验，假设我们在这里处理的表有 10 TB 大。 PostgreSQL 将看到INSERT操作并在此表中寻找一些备用位置（使用现有块或添加新块）。出于本示例的目的，我们只是将数据放入表的第二个块中。

只要服务器真正在交易中存活下来，一切都会好起来的。如果有人在只写了abc而不是整个数据后拔下插头会怎样？

我们必须确保 PostgreSQL 在任何给定时间点都能在中断中幸存下来，而不会丢失或损坏数据。保护您的数据不是一件好事，而是绝对必须的。

假设我们在编写 XLOG 时崩溃了。在这种情况下会发生什么？好吧，在这种情况下，最终用户会知道交易没有成功，所以无论如何他或她都不会依赖交易的成功。一旦 PostgreSQL 启动，它就可以通过 XLOG 并重放所有必要的内容以确保 PostgreSQL 处于一致状态。所以，如果我们没有通过 WAL 写入来实现，那么就会发生一些令人讨厌的事情，我们不能指望写入成功。WAL 条目将始终知道它是否完整。每个 WAL 条目内部都有一个校验和，因此 PostgreSQL 可以立即检测到问题，以防有人试图重放损坏的 WAL。这在崩溃期间尤其重要，因为我们可能无法再依赖写入磁盘的最新数据。WAL 将在崩溃恢复期间自动解决这些问题。如果 PostgreSQL 配置正确，崩溃在任何时间点都是绝对安全的。

混合读取和写入

请记住，在本节中，所有内容都是关于理解写入，以确保我们的最终目标，即对复制的全面和深入理解能够实现。因此，我们必须看看读和写是如何结合在一起的。让我们看看写入和读取如何一起进行：

  1. 写进来。
  2. PostgreSQL 将写入事务⽇志以确保可以达到一致性。
  3. PostgreSQL 将在 PostgreSQL 共享缓冲区中抓取一个块并进行记忆中的变化。
  4. 读进来。
  5. PostgreSQL 将查阅缓存并查找所需的数据。
  6. 将获得缓存命中并提供查询。

这个例子有什么意义？好吧，您可能已经注意到，我们从未讨论过实际写入基础表。我们讨论了写入缓存、XLOG 等，但从未讨论过真正的数据文件。在这个例子中，我们写入的行是否在表中是完全不相关的。原因很简单：如果我们需要一个刚刚修改过的块，无论如何我们都不会将它添加到基础表中。重要的是要了解数据通常不会在写操作之后或期间直接发送到数据文件。稍后大量写入数据以提高效率是非常有意义的。这一点之所以重要，是因为它对复制有微妙的影响。数据文件本身没有价值，因为它不一定完整也不正确。要运行 PostgreSQL 实例，您将始终需要数据文件和事务⽇志。否则，没有办法在崩溃中幸存下来。从一致性的角度来看，共享缓冲区在这里是为了完成用户对数据的查看。如果某些东西不在表中，那么它在逻辑上必须在内存中。如果发生崩溃，内存将丢失，因此会查询并重放 XLOG 以将数据文件再次变成一致的数据存储。在任何情况下，数据文件都只是故事的一半。

在 PostgreSQL 9.2 及之前的版本中，共享缓冲区专门位于 SysV/POSIX 共享内存或 Windows 上的模拟 SysV 中。 PostgreSQL9.3（撰写本文时未发布）开始使用内存映射文件，这在 Windows 下速度快得多，在 Linux 下性能没有差异，但在 BSD 下速度较慢。 BSD 开发人员已经开始修复这个问题。迁移到 mmap 是为了使配置更容易，因为 mmap 不受操作系统的限制，只要有足够的 RAM，它就是无限的。 SysVshmem 是有限的，并且通常只有在对操作系统进行相应调整的情况下才能分配大量的 SysVshmen。共享内存的默认配置因 Linux 发行版而异。Suse 倾向于更宽松一点，而 RedHat、Ubuntu 和其他一些则倾向于更保守。

在本章中，您已经了解到 PostgreSQL 的事务⽇志包含了对数据库所做的所有更改。事务⽇志本身被打包成漂亮且易于使用的 16 MB 段。使用这组更改来复制数据的想法并不牵强。事实上，这是每个关系（甚至可能是非关系）数据库系统开发过程中的一个合乎逻辑的步骤。对于本书的其余部分，您将看到如何以多种不同方式使用、获取、存储、复制和分析 PostgreSQL 事务⽇志。在大多数复制系统中，PostgreSQL 事务⽇志是整个体系结构的支柱（对于同步和异步复制）。

在不考虑一致性的情况下深入研究 PostgreSQL 事务⽇志是不可能的。在本章的第一部分，我们已经尽力从总体上解释了事务⽇志的基本概念。您已经了解到，如果没有事先记录更改的能力，就很难甚至不可能保持数据文件的良好状态。到目前为止，我们主要谈论的是损坏。由于数据文件中的条目损坏而丢失数据文件绝对不是一件好事，但损坏并不是您必须关心的唯一问题。另外两个重要的主题是：

一直到磁盘

要了解与数据丢失和一致性相关的问题，我们必须了解一大块数据是如何发送到磁盘的。下图说明了这是如何工作的：

当 PostgreSQL 想要读取或写入一个块时，它通常必须经过几层。当一个块被写入时，它将被发送到操作系统。操作系统会缓存数据并对数据进行一些操作。在某些时候，操作系统将决定数据传递给一些较低的等级。这可能是磁盘控制器。磁盘控制器将再次缓存、重新排序和按摩写入，最后将其传递到磁盘。在磁盘内部，在数据最终到达真正的物理存储设备之前，可能还有一个缓存级别。

在我们的示例中，我们使用了四层。在许多企业系统中，甚至可以有更多层。想象一下虚拟机，通过网络安装存储，例如 SAN、NAS、NFS、ATA‑over_Ethernet、iSCSI 等。许多抽象层将传递数据，并且它们中的每一个都将尝试进行优化。

但是，如果用户发出了成功返回的COMMIT语句，情况就完全不同了。无论发生什么，用户都希望提交的数据可用。用户期望成功的写入在意外重启后可用。所谓的 ACID 标准也需要这种持久性。在计算机科学中，ACID（原子性、一致性、隔离性、持久性）是一组保证数据库事务得到可靠处理的属性。从内存到磁盘为确保内核将数据从内存传递到磁盘，PostgreSQL 必须采取一些预防措施。在COMMIT 上，将发出系统调用，强制将数据写入事务⽇志。PostgreSQL 此时不必强制将数据写入数据文件，因为我们始终可以从 XLOG 中修复损坏的数据文件。如果数据安全地存储在 XLOG 中，则可以认为该事务是安全的。

关于电池的一句话

大多数生产服务器将使用 RAID 控制器来管理磁盘。这里的重点是磁盘刷新和性能通常与 RAID 控制器密切相关。如果 RAID 控制器没有电池（通常是这种情况），那么刷新时间会非常长。 RAID 控制器必须等待最慢的磁盘返回。但是，如果有电池可用，RAID 控制器可以假设电源恢复后断电不会阻止已确认的磁盘写入完成。因此，控制器可以缓存写入并简单地假装刷新。因此，一个简单的电池可以轻松地将flush性能提高十倍。

请记住，我们在本节中概述的是常识。但是，每个硬件都是不同的。我们强烈建议您检查并了解您的硬件和 RAID 配置，以了解如何处理刷新。

除了 fsync()

fsync()不是唯一的将数据刷新到磁盘的系统调用。根据您使用的操作系统，可以使用不同的刷新调用。在 PostgreSQL 中，您可以通过更改wal_sync_method来决定您首选的刷新调用。同样，可以通过调整postgresql.conf 来进行此更改。

可用的方法有open_datasync, fdatasync, fsync, fsync_writethrough, and open_sync。

如果您想更改这些值，我们强烈建议您查看您正在使用的操作系统的联机帮助页，以确保您做出了正确的选择。

PostgreSQL 一致性级别

确保一致性和防止数据丢失的成本很高，每次磁盘刷新都是昂贵的，我们在刷新到磁盘之前应该三思而后行。为了给用户选择，PostgreSQL 提供了不同级别的数据保护。这些不同的选择由两个基本参数表示，可以在postgresql.conf 中找到：

  1. fsync
  2. synchronous_commit

如果使用fsync ， fsync参数将控制数据丢失。在默认配置中，PostgreSQL 总是将提交刷新到磁盘。但是，如果fsync关闭，则根本无法保证COMMIT会在崩溃中幸存下来。数据可能会丢失，甚至可能会损坏数据。为了保护您的所有数据，有必要保持fsync开启。如果您可以承受丢失部分或全部数据的后果，则可以稍微放宽刷新标准。

synchronous_commit与 XLOG 写入有关。通常，PostgreSQL 会等到数据完全写入 XLOG。特别是空头交易可能会受到很大影响，因此提供了各种不同的选择：

  • on: PostgreSQL 将等到XLOG 完全成功写入。如果您正在存储信用卡数据，您希望确保金融交易不会丢失。在这种情况下，刷新到磁盘是必不可少的。.
  • off: 向客户端报告成功和安全写入磁盘之间会有时间差。在这样的环境中，可能会出现腐败。让我们假设一个数据库存储有关网站上当前在线用户的信息。假设您的系统崩溃并在 20 分钟后恢复。你真的关心你的数据吗？ 20 分钟后，无论如何每个人都必须重新登录。无论如何，为了保护几分钟后就会过时的数据而牺牲性能是不值得的。
  • local: 在复制数据库实例的情况下，我们将只等待本地实例刷新到磁盘。这里的好处是你有一个高级别的保护，因为你刷新到一个磁盘；但是，我们可以安全地假设两台服务器不会同时崩溃，因此我们可以稍微放宽从服务器的标准。
  • remote_write: PostgreSQL 会等待一个同步的备用服务器报告给定交易的成功。

在将fsync设置为关闭的合同中，将synchronous_commit更改为关闭不会导致损坏。但是，在发生崩溃的情况下，我们可能会丢失一些已经成功提交的事务。潜在数据丢失量由名为wal_writer_delay的附加postgresql.conf设置控制。在将synchronous_commit设置为 off 的情况下，我们丢失的数据永远不会超过wal_writer_delayconfig变量中定义的数据。

更改 synchronous_commit 可能看起来像是一个小的性能调整；然而，实际上，更改同步行为是运行小型写入事务时的主要因素之一。

收益可能不仅仅是几个百分点，如果幸运的话，它可能是十倍甚至更多（取决于硬件、工作负载、I/O 子系统等）。请记住，配置数据库不仅仅与速度有关。一致性至少与速度一样重要，因此您应该仔细考虑是否要以速度换取潜在的数据丢失。

充分理解本章中概述的那些与一致性相关的主题非常重要。在决定您的集群架构时，数据安全将是必不可少的部分，并且非常希望能够判断某些架构是否对您的数据有意义。毕竟，数据库工作就是保护数据。充分了解您的耐用性要求绝对是一大优势。

调整检查点和 XLOG

到目前为止，本章有望提供一些关于 PostgreSQL 如何写入数据以及 XLOG 的一般用途的见解。鉴于这些知识，我们现在可以继续前进并了解我们可以做些什么来使我们的数据库更有效地工作，无论是在复制的情况下还是在仅运行单个服务器的情况下。

了解检查点在本章中，我们已经看到数据必须先写入 XLOG，然后才能到达任何地方。问题是，如果 XLOG 从未被删除，很明显，我们不会在某个时间点不填满磁盘而永远写入它。要解决此问题，必须在某个时候删除 XLOG。这个过程称为检查点。

这个问题引起的主要问题是：XLOG 什么时候可以截断到某个点？答案是：当 PostgreSQL 将 XLOG 中已有的所有内容放入存储文件时。如果对 XLOG 所做的所有更改也对数据文件进行，则 XLOG 可能会被截断。请记住，简单地写入数据是没有价值的，我们还必须将数据刷新到数据表中。在某种程度上，XLOG 可以看作是数据文件的修理工，以防万一。如果一切都已完全修复，则可以安全地删除修复说明，这正是检查点期间发生的事情。

配置检查点

检查点对于一致性非常重要，但它们也与性能高度相关。如果检查点配置不当，您可能会面临严重的性能下降。在配置检查点时，以下是相关的参数。请注意，所有这些参数都可以在postgresql.conf 中更改：

checkpoint_segments = 3 
checkpoint_timeout = 5min 
checkpoint_completion_target = 0.5 
checkpoint_warning = 30s

关于段和超时

checkpoint_segments和checkpoint_timeout将定义两个检查点之间的距离。当我们用完段或时间结束时，就会发生检查点。请记住，一个段通常为 16 MB，因此三个段意味着我们将每 48 MB 执行一次检查点。在现代硬件上，16 MB 是远远不够的。在典型的生产系统上，256 个甚至更高的检查点间隔是完全可行的。但是，在设置checkpoint_segments 时，必须牢记一件事：如果发生崩溃，PostgreSQL 必须重放自上一个检查点以来的所有更改。如果两个检查点之间的距离异常大，您可能会注意到失败的数据库实例需要很长时间才能重新启动。为了可用性，应该避免这种情况。崩溃后总是需要在性能和恢复时间之间进行权衡；你必须相应地平衡你的配置。

checkpoint_timeout也高度相关。它是两个检查点之间允许的时间上限。在保持时间不变的情况下，无限增加checkpoint_segments是没有意义的。在大型系统上，增加checkpoint_timeout已被证明对许多人有意义。

在 PostgreSQL 中，您会发现周围有固定数量的事务⽇志文件。与其他数据库系统不同，XLOG 文件的数量与事务的最大大小无关；一笔交易很容易就会比两个检查点之间的距离大很多。

写还是不写？

我们在本章中了解到，在COMMIT时，我们无法确定数据是否已经在数据文件中。因此，如果数据文件无论如何都不必保持一致，为什么不改变写入数据的时间点呢？这正是我们可以用checkpoint_completion目标做的。这个想法是有一个设置来指定检查点完成的目标，作为两个检查点之间总时间的一小部分。

现在让我们讨论三个场景来说明checkpoint_completion_target 的目的：

场景 1 – 存储股票市场数据

在这个场景中，我们想要存储道琼斯工业平均指数(DJIA)中所有股票的最新股票报价。我们不想存储所有股票价格的历史记录，而只是存储最近的当前价格。鉴于我们正在处理的数据类型，我们可以假设我们将有一个由UPDATE语句指定的工作负载。会发生什么？ PostgreSQL 必须一遍又一遍地更新完全相同的数据。考虑到 DJIA 仅包含 30 只不同的股票，数据量非常有限，我们的表格将非常小。除此之外，价格可能每秒更新一次，甚至更频繁。在内部，情况是这样的：当第一个UPDATE出现时，PostgreSQL 会抓取一个块，将其放入内存并进行修改。每个后续更新很可能会更改同一个块。从逻辑上讲，所有写入都必须转到事务⽇志，但是共享缓

冲区中的缓存块会发生什么情况？

一般规则是：如果有很多更新（分别对同一个块进行更改），明智的做法是将块尽可能长时间地保留在内存中；这将大大增加通过一次写入多个更改来避免 I/O 的几率。如果您想增加在一个磁盘 I/O 中进行多次更改的几率，请考虑降低 checkpoint_complection_target。块将在内存中停留更长时间，因此在写入发生之前，许多更改可能会进入相同的块。在刚刚概述的场景中，0.05（或 5%）的checkpoint_completion_target可能是合理的。

在我们的第二个场景中，我们将 1 TB 的数据加载到一个空表中。如果一次加载如此多的数据，再次击中 10 分钟前击中的块的几率是多少？几率基本为零。在这种情况下缓冲写入没有意义，因为我们只会错过因空闲和等待 I/O 发生而丢失的磁盘容量。在批量加载期间，我们希望使用一直拥有的所有 I/O 容量。为了确保 PostgreSQL 立即写出数据，我们必须将checkpoint_completion_target增加到接近 1 的值。

尖锐的峰值可能要了你的命；至少他们可以造成应该避免的严重伤害。在您周围的现实世界中是真实的，在数据库世界中也始终是真实的。在这种情况下，我们要假设一个应用程序存储所谓的电话公司的呼叫详细记录(CDR) 。你可以想象会发生大量的写作，人们整天都在打电话。当然，会有人打一个电话，然后立即打下一个电话，但我们也会看到很多人一周左右只打一个电话。从技术上讲，这意味着共享内存中最近被更改的块很有可能很快会面临第二次或第三次更改，但是，我们还将对不会发生的块进行大量更改再次访问过。我们该如何处理？嗯，晚写出数据是个好主意，这样尽可能多的更改将转到之前修改过的页面。

调整 WAL 缓冲区

在本章中，我们已经对shared_buffers、 fsync等主要参数进行了调整。然而，还有一个参数会对性能产生巨大影响。 wal_buffers参数的目的是告诉 PostgreSQL 需要保留多少内存来记住 XLOG，到目前为止还没有写入磁盘。因此，如果有人输入了一个大事务，PostgreSQL 将不会在 COMMIT 之前将对表的任何微小更改写入 XLOG 。请记住，如果未提交的事务在崩溃期间丢失，我们无论如何都不会关心它，因为COMMIT是⽇常生活中唯一真正重要的事情。在COMMIT发生之前以更大的块写入 XLOG 是非常有意义的。这正是wal_buffers所做的：除非在postgresql.conf 中手动更改，否则它是一个自动调整的参数（由‑1 表示） ，它使 PostgreSQL 占用shared_buffers的 3%但不超过 16 MB 以在将 XLOG 写入之前保留它磁盘。

在旧版本的 PostgreSQL 中，此参数为 64 KB。这对于现代机器来说低得不合理。如果您运行的是旧版本，请考虑将 wal_buffers 增加到 16 MB。对于大小合理的数据库实例，这通常是一个不错的值。XLOG 的内部结构我们将在整本书中使用事务⽇志，并且为了让您更深入地了解事物在技术层面上的工作原理， 我们添加了专门处理 XLOG 机制内部工作的部分。我们将避免深入到 C 级别，因为这超出了本书的范围，但我们将为您提供足够深入的见解。

了解 XLOG 记录

对 XLOG 所做的更改是基于记录的。那是什么意思？假设您要向表中添加一行：

test=# INSERT INTO t_test VALUES (1, 'hans');
INSERT 0 1

在这个例子中，我们插入到一个包含两列的表中。为了这个例子，我们假设两列都被索引了。记住我们之前学到的知识：XLOG 的目的是保证这些数据文件的安全。因此，此操作将触发一系列 XLOG 条目。首先，数据文件相关的表都会写。然后将创建索引的相关条目。最后一条COMMIT记录被发送到⽇志。

并非所有 XLOG 记录都是相同的：存在各种类型的 XLOG 记录（例如，heap、btree、clog、storage、gin 和standby records，仅举几例）。XLOG 记录是反向链接的。因此，每个条目都指向文件中的前一个条目。这样，一旦找到指向前一个条目的指针，我们就可以完全确定我们已经找到了记录的结尾。使 XLOG 具有确定性如您所见，单个更改可以触发大量 XLOG 条目。各种陈述都是如此。例如，一个大的DELETE语句很容易导致一百万次更改。原因是 PostgreSQL 不能简单地将 SQL 本身放入⽇志中；它确实必须记录对表所做的物理更改。这是为什么？只需考虑以下示例：

test=# DELETE FROM t_test WHERE id > random();
DELETE 5335

random()函数每次调用时都必须产生不同的输出，因此我们不能只将 SQL 放入⽇志中，因为如果在重放期间执行它不能保证为我们提供相同的结果。

使XLOG可靠XLOG本身是整个数据库实例中最关键和最敏感的部分之一。因此，我们必须特别小心，确保尽一切可能保护它。在崩溃的情况下，如果周围没有 XLOG，数据库实例通常就注定要失败。在内部，PostgreSQL 采取了一些特殊的预防措施来处理 XLOG：

  • 使用 CRC32 校验和
  • 禁用信号
  • 空间分配

首先，每个 XLOG 记录都包含一个 CRC32 校验和。这允许我们在启动时检查⽇志的完整性。崩溃前的最后一次写操作不再完全正常是完全可行的，因此校验和绝对可以帮助立即解决问题。校验和由 PostgreSQL 自动计算，用户不必显式处理此功能。

除了校验和之外，PostgreSQL 还会在写入 XLOG 时暂时禁用信号。这提供了一些额外的安全级别，并减少了某个地方出现愚蠢的极端情况问题的可能性。最后，PostgreSQL 使用固定大小的 XLOG。 XLOG 的大小由检查点段以及checkpoint_completion_target 确定。

PostgreSQL事务⽇志的大小计算如下：

(2 + checkpoint_completion_target) * checkpoint_segments + 1

另一种计算大小的方法是：

checkpoint_segments + wal_keep_segments + 1 files

重要的是，如果某个东西的大小是固定的，它很少会用完空间。

在基于事务⽇志的复制的情况下，如果无法归档事务⽇志，我们可能会用完 XLOG 目录上的空间。

LSN 和共享缓冲区交互

如果你想修理一张表，你必须确保按照正确的顺序进行；如果一行在它实际存在之前被删除，那将是一场灾难。因此 XLOG 为您提供了所有更改的顺序。在内部，此顺序通过逻辑序列号(LSN) 反映。 LSN 对 XLOG 至关重要。每个 XLOG 条目将立即分配给一个 LSN。在前面的一节中，我们讨论了一致性级别。将synchronous_commit设置为off，即使 XLOG 记录尚未刷新到磁盘，客户端也会得到一个 okay。尽管如此，由于更改必须反映在缓存中，并且由于 XLOG 必须在数据表之前写入，系统必须确保并非共享缓冲区中的所有块都可以立即写

出。 LSN 将保证只有当相应的更改已经写入 XLOG 时，我们才能将块从共享缓冲区写入数据文件。写入 XLOG 是基础，违反此规则肯定会在崩溃后导致问题。

了解 PITR 的目的

PostgreSQL 提供了一个名为pg_dump的工具来备份数据库。基本上， pg_dump将连接到数据库，读取事务隔离级别“可序列化”中的所有数据，并将数据作为文本返回。当我们使用“可序列化”时，转储总是一致的。因此，如果您的pg_dump在午夜开始并在早上 6 点结束，您将创建一个备份，其中包含截至午夜的所有数据，但没有更多数据。这种创建快照的方式非常方便，对于中小数据量来说是完全可行的。

转储始终是一致的。这意味着所有外键都完好无损；启动转储后添加的新数据将丢失。这很可能是执行标准备份的最常用方法。但是，如果您的数据非常有价值并且可能非常大以至于您想增量备份它怎么办？对某些应用程序来说，时不时拍一张快照就足够了；对于高度关键的数据，显然不是。除此之外，以文本形式重放 20 TB 数据的效率也不高。 Point‑In‑Time‑Recovery 旨在解决此问题。它是如何工作

的？基于数据库的快照，稍后将重播 XLOG。这可以无限期地发生，也可以达到您选择的某个点。这样，您可以到达任何时间点。

此方法为许多不同的方法和功能打开了大门：

  • 将数据库实例恢复到给定的时间点
  • 创建备用数据库，其中包含原始数据的副本
  • 创建所有更改的历史记录

在本章中，我们将特别介绍增量备份功能，并描述如何通过将更改增量归档到所选介质来使数据更加安全。

转向更大的图景

下图概述了用于时间点恢复

我们在前一章中已经看到 PostgreSQL 会生成 16 MB 的事务⽇志段。每次这些段中的一个被填满并准备就绪时，PostgreSQL 将调用所谓的archive_command。 archive_command的目标是将 XLOG 文件从数据库实例传输到存档。在我们的图像中，存档表示为图像右下角的罐子。该设计的美妙之处在于您基本上可以使用任意 shell 脚本来归档事务⽇志。这里有一些想法：

  • 使用一些简单的副本将数据传输到 NFS 共享
  • 运行rsync以移动文件
  • 使用自定义脚本对 XLOG 文件进行校验和并将其移动到一个FTP服务器
  • 将 XLOG 文件复制到磁带

管理 XLOG 的可能选项仅受想象力的限制。restore_command与archive_command完全对应。它的目的是从存档中获取数据并将其提供给实例，该实例应该重放它（在我们的图像中，这被标记为恢复备份）。如您所见，重播可用于复制或仅将数据库恢复到本章概述的给定时间点。同样， restore_command只是一个 shell 脚本，可以按您的意愿逐个文件执行任何操作。

值得一提的是，全能的管理员负责存档。您必须决定要保留多少 XLOG 以及何时删除它。这项任务的重要性不可低估。

请记住，当archive_command由于某种原因失败时，PostgreSQL 将保留 XLOG 文件并在几秒钟后重试。如果归档从某个点开始不断失败，则可能会发生 master 已满的情况。 XLOG 文件的顺序不能被打断；如果缺少单个文件，则无法继续重播 XLOG。所有 XLOG 文件都必须存在，因为 PostgreSQL 需要不间断的 XLOG 文件序列；如果单个文件丢失，恢复过程最晚会停止。

归档事务⽇志

在了解了大局之后，我们可以看看如何将事情付诸实践。当涉及到时间点恢复时，您要做的第一件事就是归档 XLOG。 PostgreSQL 通过 postgresql.conf 提供与归档相关的所有配置选项。让我们一步一步地看看在postgresql.conf中必须做什么才能开始归档：

  1. 首先，你应该打开archive_mode 。

  2. 在第二步中，您应该配置您的存档命令。 archive命令是一个简单的 shell 命令，有两个参数：

    %p：这是一个占位符，表示应存档的 XLOG 文件，包括其完整路径（来源）。

    %f：此变量保存 XLOG 的名称，但没有指向它的路径。

现在让我们设置存档。为此，我们应该创建一个地方来放置 XLOG。理想情况下，XLOG 不存储在与要存档的数据库实例相同的硬件上。为了这个例子，我们假设我们想要将存档应用到/archive。必须对postgresql.conf 进行以下更改：

wal_level = archive # minimal, archive, or hot_standby
                    # (change requires restart)
archive_mode = on	# allows archiving to be done
                    # (change requires restart)
archive_command = 'cp %p /archive/%f' 	# command to use to archive a logfile segment
                                         # placeholders: %p = path of file to archive
                                         # %f = file name only

完成这些更改后，归档就可以进行操作了，您只需重新启动数据库即可激活这些内容。在我们重新启动数据库实例之前，我们希望您将注意力集中在wal_ 级别上。目前可以使用三种不同的wal_level设置：

  • minimal
  • archive
  • hot_standby

在只有单个节点的情况下产生的事务⽇志量远远不足以同步整个第二个实例。 PostgreSQL 中有一些优化，允许在单节点模式下跳过 XLOG 写入。以下指令可以受益于将 wal_level设置为最小值： CREATE TABLE AS、 CREATE INDEX、 CLUSTER和COPY （如果表是在同一事务中创建或截断的）。要重放事务⽇志，至少需要存档。

archive和hot_standby之间的区别在于archive不必知道当前正在运行的事务。然而，对于流复制，此信息至关重要。重启可以直接通过 pg_ctl –D /data_ directory –m fast restart 完成，也可以通过标准的初始化脚本完成。检查我们的归档是否有效的最简单方法是在数据库中创建一些无用的数据。以下代码片段显示可以轻松制作一百万行：

进行基础备份

在上一节中，您已经看到启用归档只需要几行代码并提供了很大的灵活性。在本节中，我们将看到如何创建所谓的基本备份，稍后可用于应用 XLOG。基本备份是数据的初始副本。请记住，XLOG 本身或多或少毫无价值。它仅在与初始基础备份结合使用时才有用。

在 PostgreSQL 中，有两个主要选项可用于创建初始基础备份：

  • 使用pg_basebackup
  • 基于传统复制/rsync的方法

以下两节将详细说明如何创建基本备份：

使用 pg_basebackup

创建现有服务器备份的第一个也是最常见的方法是运行名为pg_basebackup 的命令，该命令在 PostgreSQL 9.1.0 中引入。

基本上pg_basebackup能够直接通过数据库连接获取数据库基础备份。当在从机上执行时， pg_basebackup将连接到您选择的数据库服务器，并将数据目录中的所有数据文件复制到您的机器上。不再需要登录盒子，只需一行代码即可运行； pg_basebackup会为你做所有剩下的事情。

在此示例中，我们假设我们要对名为sample.postgresql‑support.de 的主机进行基本备份。必须执行以下步骤：

  • 修改pg_hba.conf以允许复制
  • 向master 发送信号以考虑pg_hba.conf更改
  • 调用pg_basebackup

修改 pg_hba.conf

要允许远程机器登录到 PostgreSQL 服务器并流式传输 XLOG，您必须明确允许复制。在 PostgreSQL 中，有一个名为pg_hba.conf 的文件，它告诉服务器允许哪些盒子使用哪种类型的凭据进行连接。可以通过pg_hba.conf 允许或简单地丢弃整个 IP 范围。

要启用复制，我们必须为每个要允许的 IP 范围添加一行。以下清单包含有效配置的示例：

# TYPE DATABASE USER ADDRESS METHOD
host replication all 192.168.0.34/32 md5

在这种情况下，我们允许来自192.168.0.34 的复制连接。 IP 范围由32标识（在我们的例子中它仅代表一台服务器）。我们决定使用 MD5 作为我们的身份验证方法。这意味着pg_basebackup必须向服务器提供密码。如果您在非安全关键环境中执行此操作，则使用信任作为身份验证方法也可能是一种选择。

如果您的系统中实际上有一个名为 replication 的数据库，会发生什么情况？基本上，将数据库设置为复制只会配置您的流式传输行为，如果您想放入处理称为复制的数据库的规则，则必须引用数据库名称，如下所示：“复制”。但是，我们强烈建议不要做这种诡计，以免造成混淆。

向主服务器发送信号一旦pg_hba.conf被更改，我们就可以告诉 PostgreSQL 重新加载配置。完全不需要重新启动数据库。我们有三个选项来使 PostgreSQL 重新加载pg_hba.conf：

  • 通过运行 SQL 命令： SELECT pg_reload_conf();
  • 通过向主机发送信号： kill –HUP 4711 （ 4711是master 的进程 ID）
  • 通过调用pg_ctl： pg_ctl –D $PGDATA reload （ $PGDATA是数据库实例的主目录）

一旦我们告诉作为数据源的服务器接受流式连接，我们就可以继续运行pg_basebackup。

还有一种方法可以在执行基本备份时查看进度条，但我们不建议使用此选项 (‑‑progress)，因为它需要 pg_basebackup 首先确定源实例的大小，这可能会很昂贵。

pg_basebackup – 自给自足的备份

通常没有 XLOG 的基础备份是毫无价值的。这是因为基础备份是在主服务器完全运行时进行的。在进行备份时，数据库实例中的那些存储文件可能已被大量修改。 XLOG 的目的是可靠地修复数据文件中的那些潜在问题。但是，如果我们想创建一个基本备份，它可以在没有（明确存档）XLOG 的情况下存在怎么办？在这种情况下，我们可以使用‑‑xlog‑method=stream选项。如果选择了这个选项， pg_basebackup 不仅会按原样复制数据，还会将在基本备份期间创建的 XLOG 流式传输到我们的目标服务器。这将为我们提供足够的 XLOG，使我们能够直接启动以这种方式制作的基本备份。它是自给自足的，不需要额外的 XLOG 文件。这不是时间点恢复，但它可以在遇到麻烦时派上用场。拥有可以立即启动的基本备份通常是一件好事，而且成本相当低。

请注意 ‑‑xlog‑method=stream 将需要两个到源服务器的数据库连接，而不仅仅是一个。在源上调整 max_wal_senders 时必须牢记这一点服务器。如果您计划使用 Point‑In‑Time‑Recovery 并且绝对不需要按原样启动备份，则可以安全地跳过 XLOG 并以这种方式节省一些空间（默认模式）。

使用传统方法创建基本备份

现在， pg_basebackup是获取数据库服务器初始副本的最常用方法。情况并非总是如此。传统上，一种不同的方法已被使用，其工作方式如下：

  • 调用SELECT SELECT pg_start_backup(‘some label’);
  • 通过rsync或任何其他方式将所有数据文件复制到远程框。
  • 运行SELECT pg_stop_backup()；

这种旧方法的主要优点是不需要打开数据库连接，也不需要在源服务器上配置 XLOG‑streaming 基础设施。一个主要优势还在于您可以使用 ZFS 快照或类似方法等功能，这有助于显着减少创建初始备份所需的 I/O 量。

一旦启动了 pg_start_backup，就没有必要着急了。很快离开备份模式是没有必要的，甚至不是特别可取的。如果您连续几天处于备份模

式，什么也不会发生。PostgreSQL 将像往常一样归档事务⽇志，用户不会面临任何不利影响。当然，不及时妥善关闭备份是个坏习惯。但是，当基本备份运行时，PostgreSQL 内部工作的方式不会改变。没有任何东西被填满，没有磁盘 I/O 延迟，或任何类似的事情。表空间问题如果您碰巧使用多个表空间，如果目标框上的文件系统布局与主框上的文件系统布局相同， pg_basebackup将很好地处理这一问题。但是，如果您的目标系统不使用相同的文件系统布局，则需要做更多的工作。在这种情况下，使用传统方式进行基本备份可能会有所帮助。

如果您使用‑‑format=t （对于 TAR），将为每个表空间提供一个 TAR 文件。

密切关注网络带宽

让我们想象一个涉及两台服务器的简单场景。每台服务器可能只有一个磁盘（没有 SSD）。我们的两个盒子可能通过 1 Gbit 链路互连。如果第二台服务器开始运行pg_basebackup，您的应用程序会发生什么情况？第二个盒子将连接，开始全速传输数据，并通过使用网络的全部带宽轻松杀死你的硬盘。在 master 上运行的应用程序可能会立即面临磁盘等待并提供糟糕的响应时间。因此强烈建议控制rsync占用的带宽，以确保您的业务应用程序有足够的备用容量（主要是磁盘，CPU 通常不是问题）。

如果您想将 rsync 限制为 20 MB/秒，您可以简单地使用 rsync ‑‑bwlimit=20000。这肯定会使基础备份的创建时间更长，但它会确保您的客户端应用程序不会遇到问题。

一般来说，我们建议在主从之间使用专用网络互连，以确保基本备份不会影响正常操作。限制带宽不能用pg_basebackup板载功能来完成。当然，您可以使用任何其他工具来复制数据并获得类似的结果。如果您将 gzip 压缩与 –‑gzip 一起使用，它可以用作隐式减速器。但是，这主要是一种解决方法。

重放事务⽇志

一旦我们为自己创建了闪亮的初始基础备份，我们就可以收集数据库创建的 XLOG 文件。时机成熟时，我们可以获取所有这些 XLOG 文件并执行我们想要的恢复过程。这按照本节中的描述工作。

执行基本恢复

在 PostgreSQL 中，整个恢复过程由一个名为recovery.conf 的文件管理。 它必须驻留在基本备份的主目录中。它在启动期间被读取，并告诉数据库服务器在哪里可以找到 XLOG 存档，何时结束重播等等。

为了帮助您入门，我们决定包含一个简单的recovery.conf示例文件来执行基本恢复过程：

restore_command = 'cp /archive/%f %p'
recovery_target_time = '2013-10-10 13:43:12'

restore_command本质上与您之前看到的archive_command完全相同。 archive_command应该将数据放入存档，而restore_command应该逐个文件地为恢复实例提供数据。同样，它是一个简单的 shell 命令或一个简单的 shell 脚本，一个接一个地提供 XLOG 块。您在这里的选择仅受想象力的限制； PostgreSQL 所做的只是检查您编写的代码的返回码，并重放脚本提供的数据。

就像在postgresql.conf 中一样，我们使用%p和%f作为占位符；这两个占位符的含义完全相同。

要告诉系统何时停止恢复，我们可以设置recovery_target_time。该变量实际上是可选的。如果没有指定，PostgreSQL 会恢复直到用完 XLOG。在许多情况下，简单地消费整个 XLOG 是一个非常可取的过程；如果发生崩溃，您希望恢复尽可能多的数据。但是，情况并非总

是如此。如果你想让 PostgreSQL 在特定的时间点停止恢复，你只需输入正确的⽇期。这里的关键部分实际上是知道你想重播 XLOG 到什么程度；在实际工作场景中，这已被证明是最难回答的问题。如果您碰巧遇到了未来的 recovery_target_time，请不要担心，PostgreSQL 将从您的 XLOG 中可用的最后一个事务开始，然后简单地停止恢复。数据库实例仍将保持一致并准备好进行操作。您无法破坏 PostgreSQL，但是，如果由于缺少 XLOG 而导致数据丢失，您可能会破坏您的应用程序。

在启动 PostgreSQL 之前，您必须在包含基本备份的目录上运行chmod 700 ，否则 PostgreSQL 会出错：

DETAIL: Permissions should be u=rwx (0700).

数据库生成的⽇志量告诉我们关于还原过程需要知道的一切，绝对值得详细研究这些信息。第一行表示PostgreSQL发现中断了，要开始恢复。从数据库实例的角度来看，基本备份看起来或多或少像是需要通过重放 XLOG 进行即时处理的崩溃；这正是我们想要的。

接下来的几行（恢复的⽇志文件…）表明我们正在重播自基本备份以来创建的一个 XLOG 文件。值得一提的是，重放过程是从第六档开始的。基础备份知道从哪里开始，所以 PostgreSQL 会自动寻找正确的 XLOG 文件。

PostgreSQL 到达第六个文件（一致恢复状态达到 0/60000B8）后显示的消息很重要。 PostgreSQL 声明它已达到一致状态。这个很重要。原因是基本备份中的数据文件实际上是按定义损坏的（请参阅我们之前关于 XLOG 的章节），但是，数据文件并没有损坏到无法修复的程度。只要我们有足够的 XLOG 来恢复，我们就很好了。如果您无法达到一致的状态，您的数据库实例将无法使用，并且如果不提供额外的 XLOG，您的恢复将无法进行。实际上，无法达到一致的状态通常表示归档过程和系统设置中的某处存在问题。如果到目前为止一切正常，就没有理由不达到一致的状态。

一旦达到一致状态，一个文件接一个文件将成功重放，直到系统最终查找00000001000000000000000B文件。问题是这个文件不是由源数据库实例创建的。从逻辑上讲，会弹出一个错误。找不到最后一个文件是绝对正常的；如果 recovery_target_time 在到达 XLOG 流末尾之前不要求 PostgreSQL 停止恢复，则预计会出现这种类型的错误。别担心，您的系统实际上没问题。您已成功将所有内容重播到恰好在错误消息之前显示的文件中。

一旦所有的 XLOG 都被消耗掉并且之前讨论的错误消息被发出，PostgreSQL 报告它能够或应该重放的最后一个事务，并启动。您现在拥有一个完全恢复的数据库实例，您可以立即连接到数据库。一旦恢复结束， recovery.conf将被 PostgreSQL 重命名为recovery.done以确保它不会在稍后某个时间重新启动新实例时造成任何损害。

XLOG 中更复杂的定位

到目前为止，我们已经恢复了 16 MB 事务⽇志块中可用的最新时刻的数据库。我们还看到您可以定义所需的恢复时间戳。但现在的问题是：如何知道恢复到哪个时间点？想象一下有人在白天删除了一张桌子。如果您无法立即轻松确定恢复时间戳怎么办？如果要恢复到某笔交易怎么办？

recovery.conf有你需要的一切。如果要replay到某个事务，可以参考recovery_target_xid。只需指定您需要的事务并配置recovery_target_inclusive以包含或不包含这个非常具体的事务。使用此设置在技术上很容易，但如前所述，要找到合适的重播位置目前并不容易。在典型设置中，找到停止恢复的合理点的最佳方法是使用pause_at_recovery_target。如果将其设置为 true，则在达到恢复点时 PostgreSQL 将不会自动转为生产实例。相反，它将等待数据库管理员的进一步指示。如果您不知道要重放多远，这将特别有用。您可以重放，登录，查看数据库有多远，更改为下一个目标时间，并继续小步重放。

您必须在 postgresql.conf 中设置 hot_standby = on 以允许在恢复期间读取。在 PostgreSQL 暂停后恢复恢复可以通过调用一个简单的 SQL 语句来完成： SELECT pg_xlog_replay_resume()。它将使实例移动到您在recovery.conf 中设置的下一个位置。一旦找到正确的位置，就可以将pause_at_recovery_target设置回 false 并调用pg_xlog_replay_resume。或者，您可以简单地使用pg_ctl –D … promote来停止恢复并使实例运行。

  • 这个解释是不是太复杂了？让我们把它归结为一个简单的列表：

  • 将restore_command添加到recovery.conf文件。

  • 将recovery_target_time添加到recovery.conf文件。

  • 在recovery.conf文件中将pause_at_recovery_target设置为true 。

  • 在postgresql.conf文件中将hot_standby设置为on 。

  • 启动要恢复的实例。

  • 在实例达到一致状态后立即连接到实例停止恢复。

  • 检查您是否已经到达您想去的地方。

    如果您不是：

    ° 更改recovery_target_time。

    ° 运行SELECT pg_xlog_replay_resume()。

    ° 再次检查并在必要时重复此部分。

请记住，一旦恢复完成并且 PostgreSQL 作为普通数据库实例启动后，（从 9.2 开始）就没有办法在以后重播 XLOG。

您当然可以始终使用文件系统快照，而不是经历这个过程。文件系统快照将始终与 PostgreSQL 一起工作，因为当您重新启动快照数据库实例时，它只会认为它之前已经崩溃并正常恢复。

清理途中的 XLOG

配置存档后，您必须存储源服务器创建的 XLOG。从逻辑上讲，这不可能永远发生。在某些时候，您真的必须摆脱这个 XLOG；为您的文件制定合理且可持续的清理政策至关重要。但是请记住，您必须保留足够的 XLOG，以便始终可以从最新的基本备份执行恢复。但是如果您确定不再需要特定的基础备份，您可以安全地清除所有比您要保留的基础备份更旧的 XLOG。

管理员如何确定要删除的内容？最好的方法是简单地看一下您的存档目录：

查看列表中间的文件名。 .backup文件已由基本备份创建。它包含有关基本备份方式的一些信息，并告诉系统在何处继续重放 XLOG。如果备份文件属于您需要保留的最旧的基础备份，您可以安全地擦除所有低于文件编号6 的 XLOG；在这种情况下，可以安全删除5号文件。在我们的例子中， 000000010000000000000006.00000020.backup包含以下信息：

.backup文件还将为您提供相关信息，例如制作基本备份的时间。那里很清楚，因此普通用户应该很容易阅读这些信息。作为一次删除所有 XLOG 文件的替代方法，也可以在回放期间清理它们。一种方法是在restore_ 命令中隐藏rm命令。虽然这在技术上是可行的，但这样做并不一定明智（如果你想再次恢复怎么办？）。

此外，您可以将recovery_end_command命令添加到recovery.conf文件中。 recovery_end_command的目标是让您在恢复结束后立即自动触发一些操作。同样，PostgreSQL 将调用一个脚本来精确地执行您想要的操作。您可以轻松滥用此设置来清理旧的 XLOG当数据库声明自己处于活动状态时。

切换 XLOG 文件

如果您要进行基于 XLOG 文件的恢复，您会看到每 16 MB 将归档一个 XLOG。如果您从未设法创建 16 MB 的更改，会发生什么情况？如果你是一家小型超市，每天只能销售 50 件商品怎么办？您的系统永远无法及时填满 16 MB。

但是，如果您的系统崩溃，则潜在的数据丢失可以看作是您最后一个未完成的 XLOG 文件中的数据量。也许这对你来说还不够好。

源数据库上的 postgresql.conf 设置可能会有所帮助。 archive_timeout告诉 PostgreSQL 至少每 x 秒创建一个新的 XLOG 文件。所以，如果你是这个小超市，你可以要求数据库每天在你回家之前不久创建一个新的 XLOG 文件。在这种情况下，您可以确定当天的数据已经安全地保存在您的备份设备上。也可以让 PostgreSQL 手动切换到下一个 XLOG 文件。

服务器提供了一个名为pg_switch_xlog()的过程来完成这项工作：

test=# SELECT pg_switch_xlog();
pg_switch_xlog
----------------
 0/17C0EF8
(1 row)

当一些重要的补丁作业完成时，或者如果你想确保某个数据块安全地存储在你的 XLOG 存档中，你可能想调用这个过程。

总结

在本章中，您了解了时间点恢复，这是一种将 PostgreSQL 数据库恢复到任何所需时间点的安全且简单的方法。 PITR 将帮助您实施更好的备份策略并使您的设置更加稳健。在下一章中，我们将扩展这个主题并转向异步复制。您将学习如何使用 PostgreSQL 事务⽇志复制整个数据库实例。
