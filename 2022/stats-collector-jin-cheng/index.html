stats collector进程

postgresql
postgresql
发布日期:   2022-01-12

PostgreSQL提供了一大堆系统视图，它们让管理员和开发者之类的人可以深入地审视系统中究竟在做什么。可以了解各种维度的统计信息的解读（table，index，database，replication，sql），锁解读信息，磁盘使用，数据库级别或者事务级别或者会话级别的活动信息统计，历史数据库活动统计报告。

1 postgresql用于统计信息的进程

stats collector process “src/backend/postmaster/pgstat.c”

postgresql统计信息的存放地（启动时，读入已存在的统计文件，或初始化0.数据库运行过程中存储在内存和temp文件。数据库关闭时保存到非易失存储）

  • 启动时：shared buffer -> “src/backend/postmaster/pgstat.c” -> PgstatCollectorMain -> pg_stat_read_statsfiles
  • postgresql.conf - stat_temp_directory，推荐配置在高速磁盘或内存文件系统中。
  • 数据库正常关闭时会把统计信息从tmp目录拷贝到$PGDATA/pg_stat目录中，确保统计信息不会丢失 -> pgstat_write_statsfiles(true,true)

由于统计信息的收集为查询执行增加了一些开销，因此可以将系统配置为收集或不收集信息。这由通常在postgresql.conf中设置的配置参数控制。（ 有关设置配置参数的详细信息。这些参数控制服务器范围的统计信息收集功能。启用统计信息收集后，可以通过pg_stat和pg_statio 系列系统视图访问生成的数据。

  • track_activities (布尔值)

    postgres=# select query,query_start from pg_stat_activity;
                      query                      |          query_start          
-------------------------------------------------+-------------------------------
 select query,query_start from pg_stat_activity; | 2022-01-22 10:50:46.686303+08
(1 row)

  • track_activity_query_size (整数)

    为pg_stat_activity指定为跟踪每个活动会话的当前存储的SQL长度。query字段。默认值为 1024。此参数只能在服务器启动时设置。

  • track_counts (布尔值)

    启用有关数据库活动的统计信息的收集（如表新增的行数，删除的行数等）。该参数默认开启，因为 autovacuum 守护进程需要收集的信息。只有超级用户可以更改此设置。

  • track_io_timing （布尔值）

    启用数据库 I/O 调用的计时。该参数默认关闭，因为它会重复查询操作系统当前时间，这可能会在某些平台上造成很大的开销。您可以使用pg_test_timing工具来测量系统上的计时开销。I/O 计时信息显示在pg_stat_database中，使用BUFFERS选项时 的EXPLAIN输出中，以及pg_stat_statements中。只有超级用户可以更改此设置。

  • track_functions (枚举)

    启用对函数调用计数和使用时间的跟踪。指定 pl仅跟踪过程语言函数，all也跟踪 SQL 和 C 语言函数。默认值为none，禁用函数统计跟踪。只有超级用户可以更改此设置。

    注意：无论此设置如何，都不会跟踪简单到可以 “内联”到调用查询中的 SQL 语言函数。

  • update_process_title (布尔值)

    每次服务器接收到新的 SQL 命令时启用更新进程标题。进程标题通常由ps命令查看，或者在 Windows 中使用Process Explorer查看。只有超级用户可以更改此设置。

  • stats_temp_directory (字符串)

    统计收集器通过临时文件将收集到的信息传输给其他PostgreSQL进程。这些文件存储在由指定的目录stats_temp_directory 参数，默认pg_stat_tmp。为了获得更好的性能， 可以将stats_temp_directory指向基于 RAM 的文件系统，从而降低物理 I/O 要求。当服务器关闭时，统计数据的永久副本存储在pg_stat 子目录中，以便可以在服务器重新启动时保留统计信息。此参数只能在postgresql.conf文件或服务器命令行中设置。

  • log_statement_stats ( boolean )
    log_parser_stats ( boolean )
    log_planner_stats ( boolean )
    log_executor_stats ( boolean )

    对于每个查询，将相应模块的性能统计信息输出到服务器日志。这是一个粗略的分析工具，类似于 Unixgetrusage() 操作系统工具。log_statement_stats报告总语句统计信息，而其他报告每个模块的统计信息。 log_statement_stats不能与任何每个模块选项一起启用。默认情况下，所有这些选项都被禁用。只有超级用户可以更改这些设置。

通常这些参数在postgresql.conf中设置，以便它们适用于所有服务器进程，但可以使用SET命令在单个会话中打开或关闭它们。（为了防止普通用户对管理员隐藏他们的活动，只允许超级用户使用SET更改这些参数。）

4 检查实时流量

pg_stat_activity，该视图的目的就是让用户有机会弄明白现在正在发生什么。

下面是该视图的结构：

postgres=# \d pg_stat_activity
           View "pg_catalog.pg_stat_activity"

------------------+--------------------------+-----------

此外，pg_stat_activity将为每个活动连接设置一行数据。读者将会看到数据库的内部对象ID（datid）、被连接的数据库名称以及服务于这个连接的进程ID（pid）。此外，PostgreSQL还将告诉用户谁在连接（usename，注意其中没有r），以及该用户的内部对象（usesysid）。

还有一个名为application_name的字段值。通常，该字段值可以由用户自由的设置。

postgres=# set application_name to myapp;
SET
postgres=# show application_name;
 application_name 
------------------
 myapp
(1 row)

其关键在于：假定有数千个连接来自于一个IP，作为一名管理员是否能知道一个特定的连接现在实际做什么？不可能凭记忆知道所有的SQL。如果客户端能友好地设置application_name参数，就跟容易地理解一个连接的真正目的。

接下来的3个列（client_），将告诉读者一个连接来自何方。PostgreSQL将显示IP地址，甚至主机名（如果被配置）。

此外，backend_start将会告诉读者一个特定连接何时被启动，xact_start表示一个事务何时被启动，接着还有query_start和state_change。在早期，PostgreSQL只会显示活动查询，在那个时期查询运行时间比现如今更长。在现代硬件上，OLTP查询可能只会消耗不到一毫秒，因此很难捕获到可能做了坏的查询。其解决方案是既显示活动查询，又显示正查看的连接之前执行的查询。

可能会看到这样的数据：

postgres=# select pid,query_start,state_change,state,query from pg_stat_activity where pid <> pg_backend_pid();
-[ RECORD 1 ]+------------------------------
pid          | 99619
query_start  | 2022-10-31 21:56:09.317783+08
state_change | 2022-10-31 21:56:09.317786+08
state        | active
query        | select pg_sleep(10);

在这种情况下，可以看到在第二个连接中正在执行pg_sleep。

一旦这个查询结束，输出将会改变。

postgres=# select pid,query_start,state_change,state,query from pg_stat_activity where pid <> pg_backend_pid();
-[ RECORD 1 ]+------------------------------
pid          | 99619
query_start  | 2022-10-31 21:56:09.317783+08
state_change | 2022-10-31 21:56:19.324919+08
state        | idle
query        | select pg_sleep(10);

该查询现在被标记为空闲（idle）。state_change和query_start之间的差是该查询执行所需的时间。

因此，pg_stat_activity将让用户对当前系统正在做什么有一个大体上的认识。新的state_change字段更有可能找出开销巨大的查询。

对于一个不好的查询，PostgreSQL提供了pg_cancel_backend和pg_terminate_backend两个函数来处理这些事情。pg_cancel_backend函数将终止查询，但是保留连接。pg_terminate_backend函数更加激进一些，它将查询和整个数据库连接全部杀死。

如果想要断开除自己会话之外所有其他的连接，可以这样做：

postgres=# select pg_terminate_backend(pid) from pg_stat_activity where pid <> pg_backend_pid();
 pg_terminate_backend 
----------------------
 t
(1 row)

如果正好被踢出了系统，将会显示下列消息：

postgres=# select pg_sleep(5);
FATAL:  terminating connection due to administrator command
server closed the connection unexpectedly

这意味着服务器在请求之前或处理过程中非正常停止，服务器连接丢失且需要重置。另外psql客户端会尝试重新连接。

temp_files和temp_bytes，这两列将告诉用户数据库是否不得不在磁盘上写入临时文件，这将不可避免地让操作变慢。临时文件使用量大的原因可能是什么？主要原因如下：

  • 过低的设置：如果work_men设置太低，就没有办法在RAM中做任何事情，因此PostgreSQL将利用磁盘。
  • 愚蠢的操作：经常会有人用相当昂贵但又毫无意义地查询来折磨他们的系统。如果在一个OLTP系统上看到很多临时文件，可以考虑检查一下昂贵地查询。
  • 索引和其他管理任务：有时，人们可能会创建索引或者运行DDL。这些操作可能会导致临时文件I/O，但在很多情况下不一定是问题。

简而言之，即便系统运行得很好，也可能出现临时文件。但是，注意关注临时文件，并且确保不会过于频繁地需要临时文件绝对是有意义的。

最后，还有两个更加重要的字段：blk_read_time和blk_write_time。默认情况下，这两个字段是空的，并且不会有数据被收集。它们的意图是让你有办法能查看在I/O上花费了多少时间，这两个字段为空的原因是参数track_io_timing默认被关闭，打开后需要调用C库中的计时函数，将导致产生更多开销。

在运行UPDATE时，PostgreSQL必须复制一行以确保ROLLBACK能正确的工作。这里HOST UPDATE的好处在于它允许PostgreSQL确保一行不需要离开一个块。行的复制也保留在同一块中，这样通常对性能有利。

使用pg_stat_user_tables的方法时检测哪些表可能需要索引。一种找到正确方法是使用下面的查询：

select schemaname,relname,seq_scan,seq_tup_read,
       seq_tup_read/seq_scan as avg,idx_scan 
from pg_stat_user_tables 
where seq_scan > 0 
order by seq_tup_read desc 
limit 25;

其思想是找出大型的表，它们被顺序扫描频繁地使用，那些表自然会出现在列表的顶部，它们会给出非常高的seq_tup_read值。它将帮助用户找出缺少索引的表。

注意：从上至下观察结果并寻找昂贵的扫描，记住顺序扫描不一定是不好的，在备份、分析语句中很自然地会出现顺序扫描，因此也不会造成任何伤害。不过，如果总是在运行大型地顺序扫描，系统的性能将会每况愈下。

一旦找到了可能缺失的索引，可以考虑简要的看看表的缓冲行为。pg_statio_user_tables将包含关于各种东西的信息，如表（heap_blks）、索引（idx_blks_）以及超尺寸属性存储技术（TOAST）的缓冲行为，最后能发现更多有关TID扫描的信息，但这通常和系统的总体性能无关：

pg_statio_all_tables视图将为当前数据库中的每个表（包括 TOAST 表）包含一行，显示有关该特定表的 I/O 的统计信息。pg_statio_user_tables和 pg_statio_sys_tables视图包含相同的信息，但过滤后分别只显示用户和系统表。

heap_blks_read bigint 从此表读取的磁盘块数
heap_blks_hit bigint 此表中的缓冲区命中数
idx_blks_read bigint 从此表上所有索引读取的磁盘块数
idx_blks_hit bigint 此表上所有索引中的缓冲区命中数
toast_blks_read bigint 从此表的 TOAST 表中读取的磁盘块数（如果有）
toast_blks_hit bigint 此表的 TOAST 表中的缓冲区命中数（如果有）
tidx_blks_read bigint 从该表的 TOAST 表索引读取的磁盘块数（如果有）
tidx_blks_hit bigint 此表的 TOAST 表索引中的缓冲区命中数（如果有）

尽管pg_statio_user_tables包含重要的信息，但通常的情况是pg_stat_user_tables更可能会为用户提供真正相关的见解（如缺失索引等）。

7 检查索引

虽然pg_stat_user_tables对找出缺失的索引很重要，但有时有必要找出实际不应存在的索引。而pg_stat_user_indexes可以找出那些无意义的索引。

pg_stat_all_indexes视图将为当前数据库中的每个索引包含一行，显示有关对该特定索引的访问的统计信息。pg_stat_user_indexes和pg_stat_sys_indexes视图包含相同的信息，但过滤后分别只显示用户和系统索引。

pg_stat_all_indexes View

Column Type Description
relid oid 此索引的表的 OID
indexrelid oid 此索引的 OID
schemaname name 此索引所在的架构名称
relname name 此索引的表名
indexrelname name 该索引的名称
idx_scan bigint 在此索引上启动的索引扫描数
idx_tup_read bigint 扫描此索引返回的索引条目数
idx_tup_fetch bigint 使用此索引的简单索引扫描获取的活动表行数

该视图告诉用户，对于每个模式中每个表上每个索引，多久会使用一次（idx_scan）。为了让这个视图丰富一点，可以用下面的SQL：

select schemaname,relname,indexrelname,idx_scan,
       pg_size_pretty(pg_relation_size(indexrelid)) as idx_size,
       pg_size_pretty(sum(pg_relation_size(indexrelid)) 
                      over(order by idx_scan,indexrelid)) as total 
from pg_stat_user_indexes 
order by 6;

这个语句输出非常有用，它不仅包含有关一个索引多久被用一次的信息，它还告诉我们为每个索引浪费了多少空间。最后，这个语句在列6中加上了所有的空间消耗。现在可以仔细检查该表，并且重新思考所有那些很少被使用的索引。关于何时删除一个索引很难给出一个通用规则，因此，需要手动检查更有意义。

pg_statio_all_indexes视图将为当前数据库中的每个索引包含一行，显示有关该特定索引的 I/O 的统计信息。pg_statio_user_indexes和pg_statio_sys_indexes视图包含相同的信息，但过滤后分别只显示用户和系统索引。通常该视图不包含能导致重大改进的信息。

索引的块读（区分未命中shared buffer和命中shared buffer的统计）

indexrelid oid 此索引的 OID
schemaname name 此索引所在的架构名称
relname name 此索引的表名
indexrelname name 该索引的名称
idx_blks_read bigint 从此索引读取的磁盘块数

8 查看bgwriter的统计信息

数据库连接在很多情况下不会直接写块到磁盘中。相反，数据由后台写入器或者checkpointer写入。

要看看数据如何被写入，可以查pg_stat_bgwriter视图，pg_stat_bgwriter视图总是只有一行，包含集群的全局数据。

pg_stat_bgwriter View

Column Type Description
checkpoints_timed bigint 已执行的计划检查点数
checkpoints_req bigint 已执行的请求检查点数
checkpoint_write_time double precision 在将文件写入磁盘的检查点处理部分中花费的总时间，以毫秒为单位
checkpoint_sync_time double precision 在文件同步到磁盘的检查点处理部分中花费的总时间，以毫秒为单位
buffers_checkpoint bigint 检查点期间写入的缓冲区数
buffers_clean bigint 后台写入器写入的缓冲区数
maxwritten_clean bigint 后台写入程序因写入太多缓冲区而停止清理扫描的次数
buffers_backend bigint 后端直接写入的缓冲区数
buffers_backend_fsync bigint 后端必须执行自己的fsync调用的次数（通常后台编写器会处理这些调用，即使后端自己进行写入也是如此）
buffers_alloc bigint 分配的缓冲区数
stats_reset timestamp with time zone 上次重置这些统计信息的时间

backend的buffer写过于频繁说明shared buffer不够用，或者是bgwrtiter的sleep time太长需要调整。

让bgwriter多干点活（bgwriter_delay、bgwriter_lru_maxpages，bgwriter_lru_multiplier）或者调大shared buffer。

9 跟踪、归档以及流

本节看到的一些与复制和事务日志归档相关的特性。第一项要检查的是pg_stat_archiver，它会告诉用户有关归档进程的信息，归档进程会将事务日志（WAL）从主服务器移动到某种备份设备：

postgres=# \d pg_stat_archiver
            View "pg_catalog.pg_stat_archiver"
       Column       |           Type           | Modifiers 
--------------------+--------------------------+-----------
 archived_count     | bigint                   | 
 last_archived_wal  | text                     | 
 last_archived_time | timestamp with time zone | 
 failed_count       | bigint                   | 
 last_failed_wal    | text                     | 
 last_failed_time   | timestamp with time zone | 
 stats_reset        | timestamp with time zone |

此外，pg_stat_archiver包含有关归档进程的重要信息。首先，，它会报告已经被归档的事务日志文件的数量（archive_count），它还知道最后一个归档的文件以及它何时被归档（last_archived_wal和last_archived_time）。

了解WAL文件的数量固然很有趣，但这不重要，可以考虑看failed_count和last_failed_wal。如果事务日志归档失败，这个视图将会告诉用户失败的最后一个文件以及是何时发生的。推荐对这个字段保持关注。如果不这样做就可能出现归档没有工作，但用户却没有得到通知的情况。

如果正在运行流复制，下面两个视图就非常重要：

pg_stat_replication视图将包含每个 WAL 发送方进程的一行，显示有关复制到该发送方的连接备用服务器的统计信息。仅列出直接连接的备用；没有关于下游备用服务器的信息。

pg_stat_replication View

Column Type Description
pid integer WAL 发送者进程的进程 ID
usesysid oid 登录到此 WAL 发送者进程的用户的 OID
usename name 登录到这个 WAL 发送者进程的用户名
application_name text 连接到此 WAL 发送者的应用程序的名称
client_addr inet 连接到这个 WAL 发送者的客户端的 IP 地址。如果此字段为空，则表明客户端通过服务器机器上的 Unix 套接字连接。
client_hostname text 已连接客户端的主机名，由client_addr的反向 DNS 查找报告。该字段仅对于 IP 连接不为空，并且仅在启用log_hostname时。
client_port integer 客户端用于与此 WAL 发送方通信的 TCP 端口号，如果使用 Unix 套接字，则为-1
backend_start timestamp with time zone 此进程开始的时间，即客户端连接到此 WAL 发送者的时间
state text 当前 WAL 发送者状态
sent_location text 在此连接上发送的最后一个事务日志位置
write_location text 此备用服务器写入磁盘的最后一个事务日志位置
flush_location text 此备用服务器刷新到磁盘的最后一个事务日志位置
replay_location text 最后一个事务日志位置重播到此备用服务器上的数据库中
sync_priority integer 此备用服务器被选为同步备用服务器的优先级
sync_state text 此备用服务器的同步状态

pg_stat_replication可以在复制体系中的发送服务器上查询，而pg_stat_wal_receiver则可以在接收端上查看。它能提供类似的信息并且允许在复制机上提取这些信息。

postgres=# \d pg_stat_wal_receiver
            View "pg_catalog.pg_stat_wal_receiver"
        Column         |           Type           | Modifiers 
-----------------------+--------------------------+-----------
 pid                   | integer                  | 
 status                | text                     | 
 receive_start_lsn     | pg_lsn                   | 
 receive_start_tli     | integer                  | 
 received_lsn          | pg_lsn                   | 
 received_tli          | integer                  | 
 last_msg_send_time    | timestamp with time zone | 
 last_msg_receipt_time | timestamp with time zone | 
 latest_end_lsn        | pg_lsn                   | 
 latest_end_time       | timestamp with time zone | 
 slot_name             | text                     | 
 conninfo              | text

首先，PostgreSQL将告诉用户WAL接受进程的ID，然后展示了使用中连接的状态。receive_start_lsn将告诉用户WAL接收者被启动时使用的事务日志位置。此外receive_start_tli包含WAL接收者被启动时使用的时间线。在某一时刻，可能想要知道最后的WAL位置及时间线。要得到这两个数字。

last_msg_send_time和last_msg_receipt_time它们说明一个消息何时被发送以及被收到。

latest_end_lsn和latest_end_time时刻报告给WAL发送者进程的最后一个事务日志位置。

10 检查SSL连接

PostgreSQL提供了一个视图来得到那些加密连接的概要，它就是pg_stat_ssl

postgres=# \d pg_stat_ssl
   View "pg_catalog.pg_stat_ssl"
   Column    |  Type   | Modifiers 
-------------+---------+-----------
 pid         | integer | 
 ssl         | boolean | 
 version     | text    | 
 cipher      | text    | 
 bits        | integer | 
 compression | boolean | 
 clientdn    | text    |

每一个进程都由它的进程ID表示。如果一个连接使用了SSL，则第二个列被设置为真，第三和第四列将定义版本以及密码。最后，还有加密算法使用的位数、是否使用压缩的指示器以及来自客户都安证书的标识名（DN）域。

pg_stat_xact_user_tables可以观察个体事务。它不包含系统范围的事务而只有关于当前事务的数据：

postgres=# \d pg_stat_xact_user_tables
View "pg_catalog.pg_stat_xact_user_tables"
    Column     |  Type  | Modifiers 
---------------+--------+-----------

因此开发者可以在事务提交前查看该事务，看看它是否导致了性能问题，它有助于把总体数据与由应用导致的数据区分开来。

应用开发者使用这个视图的最理想方式是在应用中增加一个函数调用，在事务提交前跟踪事务做了什么。

12 跟踪清理进度

在PostgreSQL9.6中，社区引入了一个很多人一直在期待的系统视图。很多年来，人们都想跟踪一个清理进程的进度，想看看还有多久才能做完。

因此pg_stat_progress_vacuum就是来回答这个问题的：

postgres=# \d pg_stat_progress_vacuum
View "pg_catalog.pg_stat_progress_vacuum"
       Column       |  Type   | Modifiers 
--------------------+---------+-----------
 pid                | integer | 
 datid              | oid     | 
 datname            | name    | 
 relid              | oid     | 
 phase              | text    | 
 heap_blks_total    | bigint  | 
 heap_blks_scanned  | bigint  | 
 heap_blks_vacuumed | bigint  | 
 index_vacuum_count | bigint  | 
 max_dead_tuples    | bigint  | 
 num_dead_tuples    | bigint  |

大部分列不言而喻。需要记住的是，处理不是线性的，它可能会比较跳跃。此外，清理操作通常都很快，因此进展可能很迅速，并且难于追踪。
