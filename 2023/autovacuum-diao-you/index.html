autovacuum调优

postgresql
postgresql
发布日期:   2023-02-22

这篇博问介绍关于autovacuum调优的基础知识，将非常简要地解释必要的理论(死元组、膨胀以及autovacuum如何处理它)，主要焦点是调优——有哪些配置选项、经验法则等等。

[TOC]

死元组

首先，让我们简要地解释什么是“死元组”和“膨胀”。

当你在PostgreSQL中执行DELETE操作时，行(又名元组)不会立即从数据文件中删除。相反，它仅通过在表头中设置xmax字段标记为已删除。UPDATE也是如此，在PostgreSQL中可以看作是DELETE + INSERT。

这是PostgreSQL MVCC背后的基本思想之一，因为它允许在不同进程之间只有最小的锁的情况下实现更高的并发性。当然，这种MVCC实现的缺点是它会留下已删除的元组，即使在所有可能看到这些版本的事务完成之后。

如果不清理，这些“死元组”(实际上对任何事务都不可见)将永远留在数据文件中，浪费磁盘空间，对于具有许多DELETE和UPDATE的表，死元组可能很容易占据绝大多数磁盘空间。当然，这些死元组也会从索引中引用，这进一步增加了浪费的磁盘空间。这就是我们在PostgreSQL中所说的“膨胀”。当然，需要处理的数据查询越多(即使99%的数据立即被视为“死亡”而丢弃)，查询就越慢。

VACUUM和autovacuum

回收死元组占用的空间(并使其可用于新行)的最直接方法是手动运行VACUUM命令。这个维护命令将扫描表并从表和索引中删除死元组——它通常不会将磁盘空间返回给操作系统，但它将使其可用于新行。

注意：VACUUM FULL会回收空间并将其返回给操作系统，但它有一些缺点。首先，它在表上获得排他锁，阻塞所有操作(包括select)。其次，它实际上创建了表的副本，将所需的磁盘空间增加了一倍，因此在磁盘空间已经用完的情况下，这种方法不太实用。

VACUUM的问题是它完全是手动操作，它只发生在你决定运行它的时候，而不是在需要它的时候。您可以将其放入cron中，并每5分钟对所有表运行一次，但大多数运行实际上不会清理任何东西，唯一的影响是系统上的CPU和I/O使用率更高。或者您可以每天晚上只运行一次，在这种情况下，您可能会积累更多您想要的死元组。

这就引出了自动vacuum的主要目的，根据需要进行清理，以控制浪费的空间量。数据库确实知道随着时间的推移产生了多少死元组(每个事务报告它删除和更新的元组的数量)，因此当表积累了一定数量的死元组时(我们将看到默认情况下是表的20%)，可以触发清理。因此，它将在繁忙期间更频繁地执行，而在数据库大部分空闲时则较少执行。

autoanalyze

清除死元组并不是autovacuum的唯一任务，它还负责更新优化器在计划查询时使用的数据分布统计信息。您可以通过运行手动收集这些信息ANALYZE，但它会遇到类似VACUUM的问题，您可能运行得太频繁或不够频繁。

解决方案也类似，数据库可以观察表中有多少行发生了变化，并ANALYZE自动运行。

注意：对ANALYZE来说，负面影响更严重一些，因为VACUUM的成本与死元组的数量成正比(所以当死元组很少或没有时相当低)，ANALYZE必须在每次执行时从头开始重建统计数据。另一方面，如果你不经常运行它，糟糕的计划选择的成本可能同样严重。

为了简洁起见，在本文的其余部分，我将基本忽略这个autovacuum任务。无论如何，配置与清理非常相似，并且遵循大致相同的推理。

监控

在进行任何类型的调优之前，您需要能够收集相关数据，否则您怎么能说您需要进行任何调优或者评估配置更改的影响呢?

换句话说，您应该有一些基本的监控，从数据库中收集指标。为了清理，你至少需要查看这些值:

  • pg_stat_all_tables.n_dead_tup ——每个表(包括用户表和系统表)中死亡元组的数量
  • (n_dead_tup / n_live_tup) ——每个表中死元组和活元组的比例
  • (pg_class.relpages / pg_class.reltuples )——“每行”的空间

如果您已经部署了一个监视系统(您应该这样做)，那么您很可能已经在收集这些指标。总体目标是获得稳定的行为，任何这些指标都没有突然/重大的变化。

还有一个方便的pgstattuple扩展，允许您对表和索引执行分析，包括计算空闲空间的数量，死元组等。

调整目标

在查看实际的配置参数之前，让我们简单地讨论一下高级调优目标是什么，即在更改参数时我们想要实现什么:

  • 清理死元组——保持合理的低磁盘空间量，不浪费不合理的磁盘空间量，防止索引膨胀并保持查询速度。

  • 尽量减少清理的影响——不要太频繁地执行清理，因为这会浪费资源(CPU、I/O和RAM)，并可能严重损害性能。

也就是说，你需要找到正确的平衡，太频繁地运行它可能和不够频繁地运行它一样糟糕。平衡在很大程度上取决于您管理的数据量，您正在处理的工作负载类型(DELETE/UPDATE的数量)。

postgresql.conf中的大多数默认值都相当保守，原因有二。首先，默认值是几年前确定的，基于当时常见的资源(CPU, RAM，…)。其次，我们希望默认配置可以在任何地方工作，包括像树莓派这样的小型机器或小型VPS服务器。然而，对于许多部署(特别是较小的部署和处理以读取为主的工作负载)，默认配置参数可以很好地工作。

随着数据库大小和写入量的增加，问题开始出现。典型的问题是清理不够频繁，当它发生时，它会严重破坏性能，因为它必须处理大量垃圾。如果是这种情况，你应该遵循以下简单的规则:

如果出现，那是因为你做得不够频繁。也就是说，调优参数，以便更频繁地进行清理，并且每次处理的死元组数量更少。

注意：人们有时遵循不同的规则——如果疼，就不要做。并完全禁用autovacuum。请不要这样做，除非你真的知道你在做什么，并且有定期的清理脚本。否则，您将陷入困境，您将不得不处理性能严重下降甚至可能停机的问题，而不是性能有所下降。

阈值和比例因子

当然，您可能调整的第一件事是何时触发清理，这受到两个参数的影响：

autovacuum_vacuum_threshold = 50
autovacuum_vacuum_scale_factor = 0.2

当死亡元组的数量超过(可以看到pg_stat_all_tables.n_dead_tup)时，就会触发清理

threshold + pg_class.reltuples * scale_factor

这张表将被认为需要清理。这个公式基本上是说，在一个表被清理之前，高达20%的表可能是死元组(50行的阈值是为了防止非常频繁地清理小表)。

默认的比例因子适用于小型和中型表，但不适用于非常大的表，在10GB的表上大约有2GB的死元组，而在1TB的表上大约有200GB。

这是一个积累大量死元组的例子，并一次性处理所有这些元组，这是很麻烦的。根据之前提到的规则，解决方案是通过显著降低比例因子来更经常地做这件事，甚至可能像这样:

autovacuum_vacuum_scale_factor = 0.01

这将限制减少到表的1%。另一种解决方案是完全放比例因子，只使用阈值

autovacuum_vacuum_scale_factor = 0
autovacuum_vacuum_threshold = 10000

它应该在生成10000个死元组后触发清理。

一个问题是，postgresql.conf中的这些更改会影响所有表(实际上是整个集群)，并且可能会不利地影响小表的清理，包括例如系统表。

当更频繁地清理小表时，最简单的解决方案是完全忽略这个问题。清理小表的成本相当低，对大表的改进通常非常显著，即使忽略小表上的小效率，总体效果仍然是非常积极的。

但是，如果你决定以一种会显著延迟小表清理的方式更改配置(例如设置scale_factor=0和threshold=10000)，最好只使用ALTER TABLE将这些更改应用到特定的表:

ALTER TABLE t SET (autovacuum_vacuum_scale_factor = 0);
ALTER TABLE t SET (autovacuum_vacuum_threshold = 10000);

尽量保持配置尽可能简单，并覆盖尽可能少的表的参数。将其包括在内部文档中也是一个好主意，包括特定值的推理。

节流

autovacuum内置的一个很好的功能是节流。清理是在后台运行的维护任务，对用户查询等的影响最小。换句话说，它不应该消耗太多资源(CPU和磁盘I/O)，这正是autovacuum内置节流的目的。

清理过程相当简单，它从数据文件中读取页面(8kB数据块)，并检查是否需要清理。如果没有死元组，页面将不做任何更改而直接丢弃。否则它将被清除(死元组将被删除)，被标记为“脏”并最终被写入。成本计算基于三个基本操作定义：

vacuum_cost_page_hit = 1
vacuum_cost_page_miss = 10
vacuum_cost_page_dirty = 20

也就是说，如果从shared_buffers中读取页面，则将其计数为1。如果在shared_buffers中没有找到它，需要从操作系统中读取，它将被计算为10(它可能仍然从RAM中提供，但我们不知道)。最后，如果页面被清理弄脏了，它被算作20。这使我们能够计算autovacuum所做的“工作成本”。

然后通过限制一次性可以完成的工作量来完成节流，默认设置为200，每次清理工作完成这么多工作时，它将休眠20毫秒:

autovacuum_vacuum_cost_delay = 20ms
autovacuum_vacuum_cost_limit = 200

那么，这实际上允许做多少功呢？在20ms的延迟下，清理每秒可以进行50轮，每轮200个token，这意味着每秒10000个token。意味着:

  • 80 MB/s从shared_buffer读取(假设它没有被脏)
  • 8MB/s从操作系统读取(可能是从磁盘)
  • 4MB/s写入(被autovacuum进程弄脏的页面)

考虑到当前硬件的能力，并且读/写大多是顺序的，这些限制太低了。

我们通常所做的是增加cost_limit参数，例如增加到1000(或2000)，这将使吞吐量增加 5 倍(或 10 倍)。当然，您可以调整其他参数(每页操作成本、睡眠延迟)，但我们很少这样做，改变成本限制就足够了。

Autovacuum Workers

尚未提及的一个配置选项是autovacuum_max_workers，那么它是关于什么的呢，好吧，清理不会在单个autovacuum进程中发生，但是数据库允许启动autovacuum_max_workers进程，这些进程实际上对不同的数据库/表进行清理。

这是有用的，因为您不希望在清理单个大表之前停止清理小表(由于节流，这可能需要相当多的时间)。

问题是用户认为工人的数量与可能发生的清理量成正比。如果你将自动清理的工人数量增加到6个，它肯定会比默认的3个工人多做两倍的工作，对吗?

嗯，没有。前几段所述的成本限制是全局的，由所有自动清理工人共享。每个工作进程只能获得总成本限制的1/autovacuum_max_workers，因此增加工作进程的数量只会使它们运行得更慢。

这有点像高速公路，将汽车数量增加一倍，但让它们以一半的速度行驶，每小时到达目的地的人数大约相同。

因此，如果您的数据库清理跟不上用户活动，那么增加工作人员数量不是解决方案，除非您还调整其他参数。

每表节流

事实上，当我说成本限制是全局的，并且是由所有自动清理工人共享的，我一直在说谎(有点)。类似于缩放因子和阈值，可以设置每个表的成本限制和延迟：

ALTER TABLE t SET (autovacuum_vacuum_cost_limit = 1000);
ALTER TABLE t SET (autovacuum_vacuum_cost_delay = 10);

然后，处理这些表的工人不包括在全局成本计算中，而会被独立控制。

这给了你相当大的灵活性和权力，但别忘了，权力越大，责任越大!

在实践中，我们几乎从不使用这个特性，有两个基本原因。首先，您通常希望在后台清理上使用一个全局限制。其次，多个工作人员有时被限制在一起，有时是独立的，这使得监视和分析系统的行为变得更加困难。

概括

这就是调整autovacuum的方法。如果我必须把它总结成几条基本规则，那就是这五条:

  • 不要禁用autovacuum，除非你真的知道你在做什么，认真对待。
  • 对于繁忙的数据库(执行大量UPDATEs和DELETE)，尤其是大型数据库，您可能应该降低比例因子，以便更频繁地进行清理。
  • 在合理的硬件(良好的存储、多核)上，您可能应该增加节流参数，以便清理工作能够跟上。
  • 在大多数情况下，仅仅增加autovacuum_max_workers并不能真正起作用，你会得到更多缓慢的进程。
  • 可以使用ALTER TABLE设置每个表的参数，但如果确实需要这样做，请三思。它使系统更复杂，更难以检查。
