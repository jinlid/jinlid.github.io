autovacuum 不是敌人

postgresql
postgresql
发布日期:   2023-02-23

PostgreSQL中大量的读写工作负载不可避免地会导致数据库效率低下，这是一个常见的误解。然而，只要配置正确，PostgreSQL可以毫无问题地处理这些工作负载。

这些问题源于所谓的“膨胀”，这是PostgreSQL和其他MVCC数据库的一种现象，会导致空间占用增加和性能下降。我们将看到autovacuum这个对抗膨胀的工具是如何被误解和错误配置的。通过对PostgreSQL内部结构的底层了解，我们将得到一个更好的autovacuum配置。

糟糕的MVCC

问题就在这里：数据库设计人员希望允许数据库中的只读事务返回结果，而不会阻塞并发更新。这样做可以减少在web应用程序中常见的重读工作负载中请求的延迟。

然而，为了允许连续读取不间断地进行，需要为某些请求维护不同的快照，并最终协调这些差异。这些善意的谎言会招致所有说谎者都熟悉的空格惩，你需要良好的记忆力来保持故事的连贯性!

PostgreSQL和其他现代关系数据库使用一种称为多版本并发控制(MVCC)的技术来跟踪每个事务的快照，MVCC的空间损失称为膨胀。PostgreSQL是一台膨胀的机器，它会毫无顾忌地创建它。PostgreSQL需要一个叫做vacuum的外部工具来帮助清理。

由于我们将在后面讨论的原因，臃肿的表和索引不仅浪费空间，而且会降低查询速度。所以这不是一个大硬盘驱动器和忘记膨胀的问题。哪里有更新，哪里就有膨胀，这取决于你是否vacuum。

情况不像以前那么糟了。在遥远的过去(PostgreSQL 8之前)，DBA必须手动清理。他们必须平衡清理的资源消耗和现有的db负载，以决定何时进行清理，以及何时可能中断清理。现在，我们可以配置autovacuum 守护进程在最合适的时候为我们清理。

autovacuum在配置正确时工作良好。但是，它的默认配置适用于几百兆字节大小的数据库，而不适用于更大的数据库。在生产环境中，它开始落后。

一旦vacuum落后，它在运行时会消耗更多的资源，并且会干扰正常的查询操作。这可能导致恶性循环，数据库管理员错误地重新配置“资源霸占者”autovacuum，使其运行频率降低或根本不运行。autovacuum不是敌人，关闭它是灾难性的。

收缩与膨胀

PostgreSQL用递增的标识符(txid)为每个新事务编号。表中的所有行都有隐藏列(xmin, xmax)，记录允许查看行的最小和最大事务id。您可以想象SELECT语句隐式地包括WHERE xmin <= txid_current()和(xmax = 0 OR txid_current() < xmax)。活动事务或未来事务无法看到的行被认为是“死行”。这意味着没有xmin ≤ txid < xmax的活动事务。

新插入或更新的行使用创建它们的事务的txid作为它们的xmin，删除的行使用删除它们的txid设置xmax。

简单的说明：

begin;

select txid_current(); -- suppose it returns 1
create table foo (bar integer);
insert into foo (bar) values (100);

select xmin, xmax from foo;

commit;

返回

┌──────┬──────┐
│ xmin │ xmax │
├──────┼──────┤
│    1 │    0 │
└──────┴──────┘

如果我们更新行，那么xmin前进：

begin;

update foo set bar = 200;
select xmin, xmax from foo;

commit;

这表明……

┌──────┬──────┐
│ xmin │ xmax │
├──────┼──────┤
│    2 │    0 │
└──────┴──────┘

这没有显示的是，现在在表中有一个死行。更新一行实际上是删除它，并插入一个具有更改值的新行。我们看到的行是新插入的(通过txid 2)，原始行挂在磁盘上，xmin=1, xmax=2。我们可以通过询问关于这个表中的元组(行)的信息来确认。

create extension pgstattuple;

select tuple_count, dead_tuple_count from pgstattuple('public.foo');
┌─────────────┬──────────────────┐
│ tuple_count │ dead_tuple_count │
├─────────────┼──────────────────┤
│           1 │                1 │
└─────────────┴──────────────────┘

PostgreSQL还提供了一个低级API来查看关于物理数据库页(存储在磁盘上的表空间块)的信息。这个API允许我们查看所有行的xmin和xmax，尽管出于安全考虑，已删除行的值是不可见的。

create extension pageinspect;

select t_xmin, t_xmax from heap_page_items(get_raw_page('foo', 0));
┌────────┬────────┐
│ t_xmin │ t_xmax │
├────────┼────────┤
│      1 │      2 │
│      2 │      0 │
└────────┴────────┘

如果autovacuum正在运行，它可以清除死行，除非…被删除的行被阻止死亡！在这种恐怖电影场景中，一个事务会运行很长时间(比如分析查询)，它的txid可以防止行被标记为死亡，即使它们被另一条语句删除了。长时间运行的查询甚至不需要查询已删除的行，查询开始时存在的行确保了它们不能被删除。混合OLTP和长时间运行的分析查询是一种危险的鸡尾酒。

除了上面棘手的僵尸末日，autovacuum可以通过适当的配置来控制事情。在考虑autovacuum之前，让我们先看看膨胀的一些后果。

膨胀和查询速度

除了浪费空间外，膨胀还会影响查询速度。每个表和索引都存储为固定大小的页面数组(通常为8KB)。当查询请求行时，数据库将这些页加载到内存中。每页死行越多，加载期间浪费的I/O就越多。例如，顺序扫描必须加载并传递所有死行。

膨胀还使查询的活动元组不太可能一次性放入内存。膨胀使得每个物理页的活动元组更稀疏，因此对于相同数量的活动行，内存中需要更多的页。这将导致交换，并使某些查询计划和算法不适合执行。

一个令人讨厌的表膨胀的例子是PostgreSQL自己的系统表。目录可能会膨胀，因为它们也是表。导致这种情况发生的一种方法是循环遍历临时表，不断地创建和销毁它们。这将导致不断更新系统表。当系统表膨胀时，管理函数会变慢，甚至像在psql中运行\d这样的事情也会变慢。

索引也会膨胀。索引是从数据键值到元组标识符的映射。这些标识符命名页的“堆”(又名表存储)和该页内的偏移量。每个元组都是一个独立的对象，需要自己的索引项。行更新总是为该行创建全新的索引项。

由于一些原因，索引的性能下降没有表的严重。指向已死元组的索引项可以标记为已死，这会使索引在大小上变得臃肿，但不会导致不必要的堆查找。此外，对堆中不影响已索引列的元组的更新使用一种称为HOT的技术来提供从死亡元组到其替代元组的指针。这允许查询通过遍历堆的指针重用旧索引项。

索引膨胀的大小考虑仍然是重要的。例如，btree索引由页的二叉树组成(与您在堆中发现的元组相同大小的页)。叶子节点页包含值和元组标识符。统一随机表更新可以使btree索引保持良好的状态，因为它可以重用页面。然而，不平衡的插入/更新会影响树的一侧，同时保留一些零散的条目，可能会导致大量的大部分为空的页面。

要查看btree索引是否有效地使用了其页空间，可以询问pgstatindex。平均叶密度为索引叶页使用率的百分比：

SELECT avg_leaf_density FROM pgstatindex('btree_index_name');

调优autovacuum

autovacuum使数据库保持整洁和快速，它在满足某些可配置条件时开始工作。当它检测到它对常规查询过于侵入时，它会暂停。

对于集群中的每个数据库，autovacuum每隔autovacuum_naptime(默认1分钟)尝试启动一个新的worker。它一次最多运行autovacuum_max_workers(默认3个)。

每个worker都在寻找一张需要帮助的表。worker搜索那些PostgreSQL的统计记录表明相对于表大小发生了有足够多的行变化的表。特别地，工作人员要查找[估计无效的行数]≥autovacuum_vacuum_scale_factor *[估计表大小]+ autovacuum_vacuum_threshold的表。

worker开始从表中删除死元组并压缩页面。当所有工作人员继续工作时，autovacuum会记录他们正在消耗的I/O“积分”。不同类型的操作计算不同的积分(值是可配置的)。当使用的积分超过autovacuum_vacuum_cost_limit时，autovacuum暂停所有工作人员autovacuum_vacuum_cost_delay毫秒。

vacuum是一场与时间的赛跑。在压缩页面时，vacuum worker扫描堆寻找死行并将它们添加到列表中。它使用列表首先删除指向这些行的索引项，然后从堆中删除这些行。如果有很多行要清理，maintenance_work_mem是有限的，那么工作人员将无法在每次传递中处理尽可能多的死行，并且将不得不浪费时间更频繁地重复这个过程。

这解释了autovacuum可能落后的一种方式：当积累了许多死行时，autovacuum没有足够的维护工作内存来快速删除它们，并且受到vacuum_cost_limit的限制。这在数据库中不成比例的大表上可能会被读出来。默认数据库值autovacuum_vacuum_scale_factor=0.2可能适用于小型表，但对于大型表来说它太大了。你可以配置每个表的参数:

ALTER TABLE <tablename>
  SET autovacuum_vacuum_scale_factor = 0.01;

对于一个百万行的表，这意味着autovacuum将在1万行失效后启动，而不是20万行。它有助于防止膨胀失控。

当需要清理的膨胀表比autovacuum_max_workers要多，并且所有的表都在继续膨胀时，autovacuum也会落后。工人们不能到所有的表。

这里有灵敏的autovacuum调节配置。当然，它们并不适用于所有的数据库，但将使您朝着正确的方向前进。

参数 PG默认值 推荐值
autovacuum max workers 3 5 or 6
maintenance work mem 64MB system ram * 3/(8 * autovacuum max workers)
autovacuum vacuum scale factor 0.2 Smaller for big tables, try 0.01
autovacuum vacuum threshold 50 Could be larger for small tables
autovacuum vacuum cost limit 200 Probably leave it alone
autovacuum vacuum cost delay 20ms Can turn it down if OK with more vacuum I/O load

密切关注

调优autovacuum之后，您可以等待并观察数据库如何响应。事实上，您可能希望在调优autovacuum之前观察一段时间的数据库，以避免过早优化。您正在寻找表和索引中的变化率和膨胀百分比。

使用这些脚本收集测量结果：pgexperts/pgx_scripts。在cron作业中执行它们，以每周跟踪进度。

分散工作

繁忙的大表有很大的膨胀潜力，这既是由于对vacuum比例因子的低敏感性，也是由于行搅动的程度。水平地把一个大表分成几个小表是很有用的，特别是如果有很多vacuum worker，因为一次只能有一个工人vacuum一张表。即便如此，运行更多的工人需要更大的维护工作内存。一种既能分割大型表又能增加运行vacuum worker容量的解决方案是使用由多个物理PostgreSQL服务器和分片表组成的分布式数据库。

这不仅仅是用户查询在分布式数据库中扩展，它也是vacuum。公平地说，如果查询在单个PostgreSQL实例上可以很好地扩展，并且膨胀是唯一的问题，那么切换到分布式系统就过度了，还有其他积极治疗急性膨胀的方法。然而，额外的vacuum能力是分布式的一个令人愉快的副作用。
