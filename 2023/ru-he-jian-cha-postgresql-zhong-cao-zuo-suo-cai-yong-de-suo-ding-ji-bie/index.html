如何检查 PostgreSQL 中操作所采用的锁定级别

开发语言
python
发布日期:   2023-02-23

PostgreSQL 的手册通常对各种操作所采用的锁非常清楚——但没有什么是完美的。如果需要，也可以直接问PostgreSQL。

您可以使用psql或 PgAdmin 简单地检查锁定级别。

比如看什么锁alter table some_table disable trigger some_trigger;需要：

test=> BEGIN;
BEGIN
test=> ALTER TABLE some_table DISABLE TRIGGER some_trigger;
ALTER TABLE
test=> SELECT locktype, mode FROM pg_locks WHERE pid = pg_backend_pid() AND relation = 'some_table'::regclass;
 locktype |         mode          
----------+-----------------------
 relation | ShareRowExclusiveLock
(1 row)

那是为了锁定一张表。从这里很容易看出我们采取了SHARE ROW EXCLUSIVE锁，根据手册：

…保护表免受并发数据更改，并且是自排他的，因此一次只有一个会话可以保存它。

pg_locks的其他用途

您还可以通过其他方式过滤pg_locks以查看事务持有的其他锁。你必须在pg_class等上做一些连接来解码关系 OID 到名称，这就是为什么我们真的需要PostgreSQL 中的pg_stat_locks视图来使这更容易。

人们使用各种不同质量的固定查询来查看当前哪些进程阻塞了其他进程。在 9.6 中添加了更详细的锁等待信息和pg_blocking_pids()，这将变得容易得多，尽管 9.6 还没有添加辅助视图。

什么是virtualxid和transactionid锁？

您将在pg_locks中看到的一个重要且可能令人困惑的事情是每个事务都持有一个特殊的锁，称为virtualxid锁：

test=> BEGIN;
BEGIN
test=> SELECT * FROM pg_locks WHERE pid = pg_backend_pid();
  locktype  | database | relation | page | tuple | virtualxid | transactionid | classid | objid | objsubid | virtualtransaction |  pid  |      mode       | granted | fastpath 
------------+----------+----------+------+-------+------------+---------------+---------+-------+----------+--------------------+-------+-----------------+---------+----------
 relation   |    16386 |    11673 |      |       |            |               |         |       |          | 2/3983             | 24250 | AccessShareLock | t       | t
 virtualxid |          |          |      |       | 2/3983     |               |         |       |          | 2/3983             | 24250 | ExclusiveLock   | t       | t
(2 rows)
test=> select '11673'::regclass;
 regclass 
----------
 pg_locks
(1 row)

如您所见，关系 AccessShareLock 只是我们在查询时对pg_locks采取的锁，因此您可以忽略它。

virtualxid锁很特殊。它是对每个交易始终持有的交易自己的虚拟交易 ID（上面的“2/3983”）的独占锁。在事务运行时，没有其他事务可以获取它。这样做的目的是让一个事务等待另一个事务使用 PostgreSQL 的锁定机制提交或回滚，并且它在内部使用。您通常不需要自己使用它，但是当您在pg_locks中看到它时，了解它是什么很有用。

事务有一个类似的条目，它获得一个真正的读/写事务 ID，其他事务可以使用它来等待它们提交或回滚：

test=> select txid_current();
 txid_current 
--------------
         2774
(1 row)
test=> SELECT * FROM pg_locks WHERE pid = pg_backend_pid();
   locktype    | database | relation | page | tuple | virtualxid | transactionid | classid | objid | objsubid | virtualtransaction |  pid  |      mode       | granted | fastpath 
---------------+----------+----------+------+-------+------------+---------------+---------+-------+----------+--------------------+-------+-----------------+---------+----------
 relation      |    16386 |    11673 |      |       |            |               |         |       |          | 2/3984             | 24250 | AccessShareLock | t       | t
 virtualxid    |          |          |      |       | 2/3984     |               |         |       |          | 2/3984             | 24250 | ExclusiveLock   | t       | t
 transactionid |          |          |      |       |            |          2774 |         |       |          | 2/3984             | 24250 | ExclusiveLock   | t       | f
(3 rows)

所以如果你曾经想知道它们是什么，现在你知道了。

当两个并发事务尝试将相同的键插入唯一索引（或主键）时，您会看到一个transactionid锁等待。在那种情况下，第二个事务必须等到第一个事务提交或回滚才能知道它是应该因错误而失败还是继续插入。
