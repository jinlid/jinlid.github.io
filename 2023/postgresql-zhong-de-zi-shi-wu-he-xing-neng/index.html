POSTGRESQL中的子事务和性能

postgresql
postgresql
发布日期:   2023-07-04

最近在排查PostgreSQL性能问题时，两次遇到了子事务的问题。所以我认为这是另一篇博客文章的好主题。

什么是子事务？

每个人都知道数据库事务。在以自动提交模式运行的 PostgreSQL 中，您必须使用BEGIN或者 START TRANSACTION跨多个语句的事务，并使用 END 或 COMMIT关闭它。如果您中止事务（或结束数据库会话而不提交），则事务内完成的所有工作都将被撤消ROLLBACK。

现在，子事务允许您回滚事务中完成的部分工作。您可以使用标准 SQL 语句在事务内启动子事务：

SAVEPOINT name;

“ name” 是子事务的标识符（没有单引号！）。您无法在 SQL 中提交子事务（它会与包含它的事务一起自动提交），但您可以使用以下命令将其回滚：

ROLLBACK TO SAVEPOINT name;

子事务的使用

子事务在较长的事务中很有用。在 PostgreSQL 中，事务内的任何错误都会中止事务：

test=> BEGIN;
BEGIN
test=> SELECT 'Some work is done';
     ?column?      
-------------------
 Some work is done
(1 row)
 
test=> SELECT 12 / (factorial(0) - 1);
ERROR:  division by zero
test=!> SELECT 'try to do more work';
ERROR:  current transaction is aborted, commands ignored until end of transaction block
test=!> COMMIT;
ROLLBACK

对于执行大量工作的事务，这是非常烦人的，因为这意味着迄今为止完成的所有工作都会丢失。子事务可以帮助您从这种情况中恢复：

test=> BEGIN;
BEGIN
test=> SELECT 'Some work is done';
     ?column?      
-------------------
 Some work is done
(1 row)
 
test=> SAVEPOINT a;
SAVEPOINT
test=> SELECT 12 / (factorial(0) - 1);
ERROR:  division by zero
test=> ROLLBACK TO SAVEPOINT a;
ROLLBACK
test=> SELECT 'try to do more work';
      ?column?       
---------------------
 try to do more work
(1 row)
 
test=> COMMIT;
COMMIT

请注意，当回滚旧事务时，ROLLBACK TO SAVEPOINT会启动另一个子事务a。

PL/pgSQL 中的子事务

即使您从未使用过该SAVEPOINT语句，您也可能已经遇到过子事务。上一节中的代码是用 PL/pgSQL 编写的，如下所示：

BEGIN
   PERFORM 'Some work is done';
 
   BEGIN  -- a block inside a block
      PERFORM 12 / (factorial(0) - 1);
   EXCEPTION
      WHEN division_by_zero THEN
         NULL;  -- ignore the error
   END;
 
   PERFORM 'try to do more work';
END;

每次您输入带有EXCEPTION子句的块时，都会启动一个新的子事务。子事务在您离开块时提交，并在您进入异常处理程序时回滚。

与其他数据库的兼容性

许多其他数据库以不同的方式处理事务内的错误。它们不会中止整个事务，而是仅回滚导致错误的语句，使事务本身保持活动状态。

当从这样的数据库迁移或移植到 PostgreSQL 时，您可能会想将每个语句包装在子事务中以模拟上述行为。

PostgreSQL JDBC 驱动程序甚至有一个连接参数“ autosave”，您可以将其设置为“ always”，以在每个语句之前自动设置一个保存点，并在失败时回滚。

如下所示，这种诱人的技术将导致严重的性能问题。

性能测试用例

为了演示过度使用子事务引起的问题，我创建了一个测试表：

CREATE UNLOGGED TABLE contend (
    id integer PRIMARY KEY,
    val integer NOT NULL
)
WITH (fillfactor='50');
 
INSERT INTO contend (id, val)
SELECT i, 0
FROM generate_series(1, 10000) AS i;
 
VACUUM (ANALYZE) contend;

该表很小，没有日志记录，并且具有较低的填充因子，以尽可能减少所需的 I/O。这样，我可以更好地观察子事务的效果。

我将使用PostgreSQL 附带的基准测试工具pgbench来运行以下自定义 SQL 脚本：

BEGIN;
PREPARE sel(integer) AS
   SELECT count(*)
   FROM contend
   WHERE id BETWEEN $1 AND $1 + 100;
PREPARE upd(integer) AS
   UPDATE contend SET val = val + 1
   WHERE id IN ($1, $1 + 10, $1 + 20, $1 + 30);
 
SAVEPOINT a;
\set rnd random(1,990)
EXECUTE sel(10 * :rnd + :client_id + 1);
EXECUTE upd(10 * :rnd + :client_id);
 
SAVEPOINT a;
\set rnd random(1,990)
EXECUTE sel(10 * :rnd + :client_id + 1);
EXECUTE upd(10 * :rnd + :client_id);
 
...
 
SAVEPOINT a;
\set rnd random(1,990)
EXECUTE sel(10 * :rnd + :client_id + 1);
EXECUTE upd(10 * :rnd + :client_id);
 
DEALLOCATE ALL;
COMMIT;

该脚本将为测试号 1 设置 60 个保存点，为测试号 2 设置 90 个保存点。它使用准备好的语句来最大限度地减少查询解析的开销。

pgbench将替换:client_id为数据库会话唯一的数字。因此，只要客户端不超过 10 个，每个客户端的UPDATEs 都不会与其他客户端的冲突，但它们会SELECT互相冲突。

子交易的实施

要了解发生了什么，我们必须了解事务和子事务是如何实现的。

每当事务或子事务修改数据时，都会为其分配一个事务 ID。PostgreSQL 在提交日志中跟踪这些事务 ID ，该日志保存在pg_xact数据目录的子目录中。

但是，事务和子事务之间存在一些差异：

  • 每个子事务都有一个包含事务或子事务（“父事务”）
  • 提交子事务不需要 WAL 刷新
  • 每个数据库会话只能有一个事务，但有很多子事务

(子)事务是给定子事务的父事务的信息持久化到数据目录的pg_subtrans子目录中。由于包含该信息的事务一旦结束就会过时，因此在关闭或崩溃期间不必保存该数据。

子交易和可见性

PostgreSQL 中行版本（“元组”）的可见性由xmin和xmax系统列决定，其中包含创建和销毁事务的事务 ID。如果存储的事务 ID 是子事务的 ID，则 PostgreSQL 还必须查阅包含（子）事务的状态来确定事务 ID 是否有效。

为了确定语句可以看到哪些元组，PostgreSQL在语句（或事务）开始时拍摄数据库快照。这样的快照包括：

  • 最大事务 ID：从该事务开始的所有内容都是不可见的
  • 拍摄快照时处于活动状态的事务和子事务的列表
  • 当前（子）事务中最早可见命令的命令号

快照是通过查看进程数组来初始化的，该数组存储在共享内存中，包含有关所有当前正在运行的后端的信息。当然，它包含后端的当前事务 ID **，并且每个会话最多可容纳 64 个非中止子事务。如果此类子事务超过 64 个，则快照将被标记为*suboverflowed***。

解释测试结果

subooverflow快照不包含确定可见性所需的所有数据，因此PostgreSQL有时不得不求助于pg_subtrans。这些页缓存在共享缓冲区中，

但是您可以看到在perf输出的SimpleLruReadPage_ReadOnly中查找它们的开销。其他事务必须更新pg_subtrans以注册子事务，您可以

在perf输出中看到它们如何与读取器竞争轻量级锁。

诊断子事务过多的问题

除了看“ perf top”之外，还有其他症状可以指出这个问题的方向：

  • 您的工作负载在单线程运行时性能良好，但在多个并发数据库会话中运行时性能很差
  • 你经常会在pg_stat_activity中看到等待事件“SubtransSLRU”(旧版本中的“SubtransControlLock”)
  • 从PostgreSQL v13开始，您可以查看监控视图 pg_stat_slru 并检查name = ‘Subtrans’行中的 blks_read 是否持续增长。这表明PostgreSQL必须读取磁盘页，因为它需要访问不再缓存的子事务。
  • 如果使用 pg_export_snapshot() 函数导出快照，则data目录的pg_snapshots子目录中的结果文件将包含“sof:1”行，表示子事务数组溢出
  • 从PostgreSQL v16开始，您可以调用函数 pg_stat_get_backend_subxact(integer) 来返回后端进程的子事务数量以及子事务缓存是否溢出

结论

子事务是一个很棒的工具，但您应该明智地使用它们。如果需要并发性，每个事务启动的子事务不要超过 64 个。

本文中显示的诊断应该可以帮助您确定您是否遇到了该问题。

找出该问题的原因可能很棘手。例如，您为 SQL 语句（可能在触发器中）的每个结果行调用的带有异常处理程序的函数启动一个新的子事务可能并不明显。
