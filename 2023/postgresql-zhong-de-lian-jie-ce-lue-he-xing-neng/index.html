POSTGRESQL中的连接策略和性能

postgresql
postgresql
发布日期:   2023-07-04

PostgreSQL 中存在三种工作方式完全不同的连接策略。如果 PostgreSQL 选择了错误的策略，查询性能可能会受到很大影响。本文介绍了连接策略、如何使用索引支持它们、它们可能会出现什么问题以及如何调整连接以获得更好的性能。

连接策略的术语

表

连接组合了两个表的数据。这样的表可以是一个表（也称为基表）或任何计划节点的结果。例如，像这样的连接中

SELECT ...
FROM a
   JOIN (b LEFT JOIN c ON ...)
      ON ...

基表a将连接到b和c的连接结果。

表也可以是索引扫描的结果。因此，当我在下面说“PostgreSQL 按顺序扫描表”时，我并不是说必须对表进行顺序扫描。只是 PostgreSQL 完整地读取了“被连接的东西”。

内表与外表的关系

任何连接的执行计划如下所示：

EXPLAIN (COSTS OFF)
SELECT * FROM a JOIN b USING (id);
 
         QUERY PLAN         
----------------------------
 Hash Join
   Hash Cond: (a.id = b.id)
   ->  Seq Scan on a
   ->  Hash
         ->  Seq Scan on b
(5 rows)

我们将连接关系的上层关系（在本例中为上面的顺序扫描a），将下层关系（从计算出的哈希值）称为连接的外表b，称为内表。

连接条件和连接键

如果将一个表的每一行与另一个表的每一行组合起来，就会得到笛卡尔积或两个关系的交叉连接。连接条件是排除其中一些组合的过滤器。编写连接条件的方法有多种，但都可以转换为

a <join type> JOIN b ON <join condition>

如果连接条件的形式为

a.col1 <operator> b.col2 [AND ...]

我们称 a.col1和 b.col2为连接键。

请注意，对于内部连接，连接条件和WHERE条件之间没有区别，但这不适用于外部连接。

Nested loop连接策略

这是所有连接策略中最简单、最通用的连接策略。

PostgreSQL 按顺序扫描外部表，对于每个结果行，它扫描内部表以查找匹配的行。

有助于nested loop连接索引

由于我们按顺序扫描外表，因此外部表上的任何索引都没有帮助。但是内部表的连接键上的索引可以大大加快嵌套循环连接的速度。

nested loop连接策略的用例

如果外部表很小，则嵌套循环连接特别有效，因为这样内部循环就不会执行得太频繁。它是具有规范化数据模型的 OLTP 工作负载中使用的典型连接策略，效率很高。如果外部表很大，则嵌套循环连接通常效率非常低，即使它们受到内部表索引的支持。

除此之外，如果没有连接条件使用=运算符，它是唯一可以使用的连接策略。因此，如果没有其他策略可以使用，它也可以作为后备策略。

Hash join策略

首先，PostgreSQL 顺序扫描内部表并构建一个哈希表，其中哈希键由所有使用=运算符的连接键组成。然后它按顺序扫描外部表并探测找到的每一行的哈希值以查找匹配的连接键。

这有点类似于嵌套循环连接。构建哈希表是一项额外的启动工作，但探测哈希比扫描内部表要快得多。

有助于hash join的索引

由于我们按顺序扫描这两个关系，因此连接条件上的索引对哈希连接没有帮助。

hash join策略的用例

如果涉及的表都不是小的，而小表对应的散列表可以放入work_mem中，那么散列关联是最好的选择。这是因为PostgreSQL会分批构建散列并将其存储在临时磁盘文件中，这会影响性能。在这种情况下，优化器通常选择不同的连接策略，如merge join。

只有当连接条件中的运算符是=时，才能在散列表中查找值，因此你至少需要一个与该运算符关联的条件。

Merge join策略

在merge join中，PostgreSQL 使用=运算符选择所有连接条件。然后，它通过连接键对两个表进行排序（这意味着数据类型必须是可排序的）。然后它迭代两个排序列表并找到匹配的条目。

有助于merge join的索引

排序键上的索引可以加速排序，因此两个表上的连接键上的索引可以加速合并连接。然而，除非可以使用仅索引扫描，否则显式排序通常更廉价。

merge join策略的用例

如果涉及的表都太大，散列无法放入work_mem，优化器通常选择merge join。因此，这是连接非常大的表的最佳策略。

与hash join一样，merge join仅在至少有一个使用=操作符的连接条件时才可行。

PostgreSQL 连接策略汇总表

nested loop hash join merge join
算法 对于每个外表行，扫描内表 从内部表构建哈希，扫描外部表，探测哈希 对表合并进行排序
有帮助的索引 内部关系的连接键上的索引 无 两个表的连接键上的索引
好的策略 外表很小 该散列表放入work_mem 两张表都很大

对查询性能的影响

选择错误的连接策略会导致性能不佳：

  • 如果优化器低估了行数，它可能会错误地选择nested loop连接。然后，它扫描内部表的次数比预期的要多，这会导致性能不佳。
  • 如果优化器高估了行数，它可能会错误地选择hash join或merge join。然后它必须完全扫描这两个表，这比在内部表上有索引的nested loop连接的性能要差得多。

在这两种情况下，错误的行计数估计都是问题的原因。因此，虽然连接可能是我们花费大部分执行时间的地方，但原因是之前发生的错误估计。

如何让PostgreSQL选择正确的连接策略

找出最好的连接策略是什么（也许 PostgreSQL 正在做正确的事情）。您可以使用以下命令暂时禁用不同的连接策略SET，该命令会更改当前数据库会话中的参数：

SET enable_hashjoin = off;
SET enable_mergejoin = off;
SET enable_nestloop = off;

请注意，您不能真正禁用嵌套循环连接，只能阻止 PostgreSQL 使用它们。如果没有=运算符的连接条件，则嵌套循环连接是唯一的方法。

调整查询通常不是一个简单直接的任务。但是，这里有一些指导方针和想法：

  • 如果由于错误估计而选择了错误的连接策略，尝试改进该估计。analyze表，或许增加default_statistics_target，看看是否有不同。尝试用更简单的 WHERE 条件重写查询，使优化器的任务更容易。
  • 尝试增加work_mem并看看是否可以获得更廉价的哈希连接。
  • 配置告诉 PostgreSQL 有关您的硬件和资源的参数：random_page_cost、effective_cache_size和effective_io_concurrency，这将使它能够正确地为索引扫描定价。
  • 您可以通过仅索引扫描来加速嵌套循环和合并联接。为此，您必须将所有必需的列添加到索引（最好使用INCLUDE子句），并确保表被足够频繁地清理。

结论——连接策略和查询调优

了解 3 种连接策略（嵌套循环连接、哈希连接和合并连接）对于任何想要了解执行计划和调整查询的人来说都是至关重要的。查询调优的艺术无法通过一篇文章来表达，但我希望我能在这里收集一些相关信息。
