1.3 TOAST 功能

通常，PostgreSQL 以 8 KB 为单位将元组存储在页中。元组不跨页存储。因此，大规模的元组不能存储在页面中。为了存储更大的元组，提供了一种称为 TOAST（超大属性存储技术）的功能。当压缩后的列数据超过大小时创建TOAST数据，这由TOAST_TUPLE_THRESHOLD决定（在编译时确定）。它还将数据存储到 TOAST 表中，直到它减少到不超过 TOAST_TUPLE_TARGET。

TOAST 表

TOAST 数据存储在单独的表（另一个文件）中，该文件由 pg_class 目录的 relfilenode 列指定。在 pg_class 目录的 reltoastrelid 列中，存储了 TOAST 表的 oid。通过从 pg_class 目录中搜索 TOAST 表 (relfilenode) 的文件名，可以识别该文件。为了加快TOAST表的查找速度，TOAST索引是用TOAST表创建的。 TOAST 表没有出现在 pg_tables 目录中。

1.4 TRUNCATE语句与文件的关系

1.5 FILLFACTOR 属性

当执行 INSERT 语句时，元组被添加到页面中。如果执行器不能再将元组存储到工作页面中，它将搜索下一个空闲页面。 FILLFACTOR 是一个属性，用于指示可以存储元组的页面大小的百分比。 FILLFACTOR 的默认值为 100 (%)。因此，元组通常无间隙地存储在页面中。也可以为索引指定 FILLFACTOR。

FILL FACTOR 从 100% 减少的好处是以页面为单位的访问性能提高，因为在使用 UPDATE 语句进行更新的情况下会使用空闲空间。另一方面，由于表使用的页数扩大，读取整个表时I/O增加。对于频繁更新的表或索引，建议降低FILLFACTOR 的值。

CREATE TABLE语句执行时的FILLFACTOR校验

建表时设置FILLFACTOR，应该写到表的WITH子句中创建表语句。要验证，请参考 pg_class 目录的 reloptions 列。

FILLFACTOR 属性的设置

postgres=> CREATE TABLE fill1(key1 NUMERIC, val1 TEXT) WITH (FILLFACTOR = 85) ;
CREATE TABLE
postgres=> SELECT relname,reloptions FROM pg_class WHERE relname='fill1' ;
relname | reloptions
--------+-----------------
fill1   | {fillfactor=85}
(1 row)

ALTER TABLE 语句执行期间的行为

要更改现有表的 FILLFACTOR 属性，请在 ALTER TABLE 语句中使用 SET 子句指定它。即使更改 FILLFACTOR 属性，现有表中的元组也不会更改。

由于 FILLFACTOR 属性的设置对现有数据的影响

postgres=> INSERT INTO fill1 VALUES (generate_series(1, 1000), 'data') ;
INSERT 0 1000
postgres=> SELECT MAX(lp) FROM heap_page_items(get_raw_page('fill1', 0)) ;
max
-----
157
(1 row)
postgres=> ALTER TABLE fill1 SET (FILLFACTOR = 30) ;
ALTER TABLE
postgres=> SELECT MAX(lp) FROM heap_page_items(get_raw_page('fill1', 0)) ;
max
-----
157
(1 row)

此示例将数据存储在表 fill1 中。使用heap_page_items函数查看页面状态时，可以看到第一页存储了157个元组。本例执行ALTER TABLE语句改变FILLFACTOR属性，然后再次确认页面上的信息，可以看出存储了相同数量的元组。

2 表空间

2.1 什么是表空间

在 PostgreSQL 数据库中，DATABASE、TABLE、INDEX 和 MATERIALIZED VIEW 存储在表空间中。创建数据库集群时，默认创建两个表空间。 pg_default 表空间供一般用户使用。 pg_global 表空间包含所有数据库共享的系统目录。如果在创建数据库时未指定表空间（TABLESPACE 子句），则使用 pg_default 表空间。

参数default_tablespace

使用参数 default_tablespace 指定表空间名称，用于对象，如作为没有 TABLESPACE 子句的 TABLE、INDEX、MATERIALIZED VIEW。该参数的设置不会影响使用 CREATE DATABASE 语句的数据库的目标。此参数的默认值为 （空字符串）。为了保存对象，使用了保存数据库的表空间。此参数不仅可以作为整个实例进行更改，还可以针对每个会话进行更改。

表2.1 未指定表空间时的对象目的地

对象 参数 default_tablespace 值
指定值 无值（空字符串）
数据库 pg_default pg_default
表 指定表空间 与数据库相同的表空间
索引 指定表空间 与数据库相同的表空间
MATERIALIZED VIEW 指定的表空间 指定表空间 与数据库相同的表空间
序列 指定表空间 与数据库相同的表空间

如果使用SET语句为每个会话指定参数default_tablespace，它会检查指定名称的表空间是否存在，但实际上并不检查是否有对象创建权限。如果使用 postgresql.conf 文件以每个实例为基础指定，则不检查指定表空间的存在。在这种情况下，使用目标数据库中的表空间，如果TABLESPACE 子句被省略。

虽然 CREATE SEQUENCE 语句语法中没有 TABLESPACE 子句，但它将是受 default_tablespace 参数影响。

参数temp_tablespaces

它指定一个表空间名称列表来创建一个临时对象。如果有多个名字指定，使用的表空间将被随机选择。

checkpointer 进程以相对较长的周期写入，而 writer 进程则在短时间内一点一点地写入更改的页面（脏缓冲区）。通过writer进程的写入，可以防止checkpoint的出现导致I/O的峰值。 writer进程的写入间隔由参数bgwriter_delay决定（默认：200ms）。延迟时间在 Linux / UNIX 平台中通过“select (2)”系统调用等待，在 Windows 环境中通过 Windows API WaitForMultipleObjects (WaitLatch src/backend/port/pg_latch.c) 等待。 (WaitLatch function in src/backend/port/win32_latch.c or src/backend/port/pg_latch.c).

write block个数的最大值由参数bgwriter_lru_maxpages决定。默认值为100，实际写块数是最近请求的块数乘以参数bgwriter_lru_multiplier的值。即使在那种情况下，这个数字也不会超过参数 bgwriter_lru_maxpages。只有当参数bgwriter_lru_maxpages 和最近需要的平均页面数估计的必要页面数大于可重用页面数时，才会进行实际写入。

6.1 块和页面

在 PostgreSQL 中，I/O 是以块为基础进行的。块的大小默认为 8 KB。为了更改值，需要更改“src/include/pg_config.h”中的“#define BLOCKSZ 8192”并重新编译它。在目前的 PostgreSQL 中，一个块包含一个页，因此块和页可以认为是一回事。一个文件中可以存储 131,072 个块（页）。当数据文件超过此大小时 (8 KB × 131,072 = 1 GB)，将创建一个新文件。

该值由 pg_config.h 文件的 RELSEG_SIZE 宏决定。在每个页面中，存储了指向页面中每个元组和页眉的指针。项目指针指示元组在页面中的位置。元组存储在从页面末尾到页面顶部的位置。另一方面，项目指针将被添加到页面的末尾，紧跟在页眉之后。

7 交易ID环绕问题

7.1 事务 ID

当一个事务在 PostgreSQL 中启动时，会发出一个唯一的事务 ID（在 txid_current 函数中获得）。事务 ID 的大小是无符号的 32 位（如 src/include/ch 中所定义）。当表的元组更新时，更新的事务 ID 存储到每个元组头中。通过此功能，PostgreSQL 可以在搜索时保持引用完整性。

事务ID的确认

可以通过指定 XMAX 和 XMIN 伪列来检查表中元组对应的事务 ID。 XMIN 列显示添加元组的事务 ID（通过 INSERT 或 UPDATE）。 XMAX 列是被删除的元组的事务 ID。因此，有效元组的 XMAX 值为零。在下面的示例中，C1 = 300 列中的 XMAX 值非零，这表示更新事务已回滚。

元组的事务 ID

postgres=> SELECT XMAX, XMIN, c1 FROM data1 ;
 xmin | xmax | c1
--------+--------+-----
507751 | 0 | 100
507752 | 0 | 200
507754 | 0 | 400
507755 | 0 | 500
507756 | 507757 | 300
(5 rows)

事务ID环绕

Transaction ID 的大小为无符号 32 位，并且会单调递增。在大型系统中，使用大量事务可能会耗尽 32 位。因此，PostgreSQL提供了一种机制来避免事务ID回绕时的问题。该机制定期将旧事务 ID 替换为特殊值（FrozenXID = 2）。

这个替换过程被称为 FREEZE 过程，通常在 VACUUM 处理中完成。当表的事务数量超过参数“autovacuum_freeze_max_age ‑ vacuum_freeze_min_age”中计算的数量时，甚至会发生 FREEZE 过程如果自动 VACUUM 无效。

自动 vacuum 使用 Visibility Map 发现处理目标块。当 FREEZE 过程完成时，自动 vacuum 过程更新 Visibility Map。当执行参数 vacuum_freeze_table_age 中指定的事务数时，它会从 Visibility Map 中搜索尚未冻结的块。事务ID，它是每个表的冻结目标，由pg_class的relfrozenxid列定义目录。该列是表中 XMIN 虚拟列的最小值（FrozenXID 除外）。pg_class目录的relfrozenxid的最小值可以在pg_database目录的datfrozenxid列中找到。

在自动 VACUUM 过程失败的情况下

如果由于某种原因未执行 FREEZE 过程，当剩余事务id到wraparound 低于 1000 万，输出以下消息。

事务回绕的警告消息

WARNING: database “postgres” must be vacuumed within 9999999 transactions
HINT: To avoid a database shutdown, execute a database-wide VACUUM in “postgres”.

而且，当剩余交易数低于100万时，下面的消息是输出，系统将停止。

事务回绕的错误消息

ERROR: database is not accepting commands to avoid wraparound data loss in
database “postgres”
HINT: Stop the postmaster and use a standalone backend to VACUUM in “postgres”.

陷入这种状态的数据库以单机模式启动，执行VACUUM语句。在下面的示例对 postgres 数据库执行 VACUUM 处理。

7.2 FREEZE 处理参数

事务ID的FREEZE操作参数如下：

  • autovacuum_freeze_max_age

    指定 pg_class.relfrozenxid 可以达到的最大年龄以防止循环的交易ID。即使自动 VACUUM 被禁用，它也会启动 VACUUM worker过程。默认值为 2 亿 (200,000,000)。最小值为 1 亿（100,000,000），最大为 20 亿（2,000,000,000）。

  • vacuum_freeze_min_age

    指定在表扫描时将事务ID替换为FrozenXID的截止年龄真空。默认值为 5000 万 (50,000,000)。可以指定autovacuum_freeze_max_age 的 0 到 50% 之间的值。

  • vacuum_freeze_table_age

    当表的 pg_class.relfrozenxid 达到这个值指定的时间时，VACUUM 会主动扫描表。默认值为 1.5 亿 (150,000,000)。该值可以指定为 0 到 10 亿 (1,000,000,000) 或最多 autovacuum_freeze_max_age 的 95%。

PostgreSQL 9.3.3 之后，增加了以下参数。有多个事务ID(MultiXactId) 和冻结的参数，但细节未经证实。

  • autovacuum_multixact_freeze_max_age
  • vacuum_multixact_freeze_min_age
  • vacuum_multixact_freeze_table_age

8.2 LIKE对索引的使用

在支持区域设置的数据库中，有一个规范，对应列的索引不是用在 LIKE 子句的前面匹配。

使用区域设置的 LIKE 搜索

postgres=> CREATE TABLE locale1(c1 varchar(10), c2 varchar(10)) ;
CREATE TABLE
postgres=> CREATE INDEX idx1_locale1 ON locale1(c1) ;
CREATE INDEX
postgres=> INSERT INTO locale1 VALUES ('ABC', 'DEF') ;
INSERT 0 1
………
postgres=> ANALYZE locale1 ;
ANALYZE
postgres=> EXPLAIN SELECT C1 FROM locale1 WHERE C1 LIKE 'A%' ;
 QUERY PLAN
-----------------------------------------------------------
Seq Scan on locale1 (cost=0.00..1790.01 rows=10 width=5)
 Filter: ((c1)::text ~~ 'A%'::text)
Planning time: 1.742 ms
(3 rows)

Execution plan变成Seq Scan，可以确认所有的tuples search都执行完了。为了避免这种情况，在执行 CREATE INDEX 语句时指定选项以执行二进制索引搜索。

CREATE INDEX 语句的选项说明

根据列的数据类型，可以指定以下选项。
