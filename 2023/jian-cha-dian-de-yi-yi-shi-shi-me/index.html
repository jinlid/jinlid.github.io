检查点的意义是什么

postgresql
数据库
发布日期:   2023-02-20

在执行大量写操作的系统上，调优检查点对于获得良好的性能至关重要。然而，检查点是我们经常发现混淆和配置问题的地方之一，那么让我带你了解一下检查点，它们做什么以及如何在PostgreSQL中调优它们。

检查点的意义是什么?

PostgreSQL是一个依赖于预写日志(WAL)的数据库，所有的更改都先写入日志(更改流)，然后才写入数据文件。这提供了持久性，因为在崩溃的情况下，数据库可以使用WAL来执行恢复，从WAL读取更改并重新应用到数据文件上。

虽然这可能会使写入量翻倍，但实际上可能会提高性能。用户只需要等待WAL(被刷新到磁盘)，而数据文件只在内存中修改，然后在后台刷新。这很好，因为WAL的写在本质上是顺序的，而对数据文件的写通常是随机的。

假设系统崩溃，数据库需要执行恢复。最简单的方法是从头开始，从头重放整个WAL。最后我们应该得到一个完整的(正确的)数据库。当然，明显的缺点是需要保留和重放整个WAL。我们经常处理的数据库不是特别大(比如几百GB)，但每天产生几TB的WAL。因此，想象一下，在运行数据库一年时，您需要多少磁盘空间来保持所有WAL，以及在恢复期间重放需要多少时间。

但是，如果数据库能够保证对给定WAL位置(日志中的偏移量)的所有更改，以及到该位置的所有数据文件更改都被刷新到磁盘中呢？然后，它可以在恢复过程中确定这个位置，并只重放WAL的剩余部分，极大地缩短了恢复时间。它还可以在“已经回放”位置之前移除WAL。

这正是检查点的作用，确保在某个时间点之前不再需要WAL进行恢复，从而减少磁盘空间需求和恢复时间。

如果你碰巧是游戏玩家，你可能熟悉检查点的概念——你的角色在游戏中通过了某个点，如果你没能击败下一个boss或掉进了炽热的熔岩湖，你就从最后一个点开始，而不是从头开始。

让我们再讨论一下另一种极端情况，执行非常频繁的检查点(比如，每秒钟左右一次)。这将允许只保留少量的WAL，并且恢复将非常快(只需要重放少量WAL)。但它也会将对数据文件的异步写入转换为同步写入，严重影响用户(例如增加“COMMIT”延迟，降低吞吐量)。

因此在实践中，您希望检查点不频繁地发生，以免影响用户，但也要足够频繁，以合理地限制恢复时间和磁盘空间需求。

触发检查点

可以触发检查点的原因如下:

  • 直接执行CHECKPOINT命令

  • 执行一个需要检查点的命令(例如:’pg_start_backup’， ‘CREATE DATABASE’，或’pg_ctl stop|restart’和其他一些)

  • 距离上一个检查点已经达到配置的时间

  • 从上一个检查点开始生成一个已配置的WAL数量(也就是“WAL用光”或“WAL满”)

前两点在这里是相当不相关的，那是罕见的，手动触发的事件。这篇博文是关于如何配置另外两个影响定期检查点的事件。

这些时间/大小限制使用两个配置选项设置:

checkpoint_timeout = 5min
max_wal_size = 1GB '(在PostgreSQL 9.5之前，这是checkpoint_segments)

使用这些(默认)值，PostgreSQL将每5分钟触发一次“CHECKPOINT”，或者当WAL在磁盘上增长到大约1GB时。

注意: max_wal_size是WAL总大小的软限制，它有两个结果。首先，数据库将尽量不超过它，但允许这样做，因此在分区上保持足够的空闲空间并监视它。其次，这不是一个“每个检查点”的限制-由于扩展检查点(稍后解释)，WAL配额被划分为2 - 3个检查点。因此，使用max_wal_size，数据库将在写入300 - 500 MB的WAL后启动一个CHECKPOINT，这取决于checkpoint_completion_target ，

默认值相当低，就像样例配置文件中的大多数其他默认值一样，即使在像树莓派这样的小系统上也能工作。

但是如何为您的系统确定好的值呢？我们的目标是不频繁或不频繁地设置检查点，我们的调优“最佳实践”包括两个步骤:

  1. 选择一个合理的checkpoint_timeout值
  2. 将max_wal_size设置得足够高，以便很少能达到

很难说checkpoint_timeout的“合理”值是多少，因为它取决于恢复时间目标(RTO)，即可接受的最大恢复持续时间是多少。

不过，这有点棘手，因为checkpoint_timeout是对生成WAL所需时间的限制，而不是直接对恢复时间的限制。不幸的是，由于种种原因，我们无法确切地说出复苏需要多长时间。WAL通常由多个进程(运行DML)生成，而恢复由单个进程执行(这种限制主要是固有的，不太可能很快消失)。这不仅会影响本地恢复，还会影响使用流复制将故障转移到备用服务器。当然，本地恢复通常发生在重新启动后，此时文件系统缓存是冷的。

但通常，默认值(5分钟)相当低，30分钟到1小时之间的值相当常见。PostgreSQL 9.6甚至将最长时间增加到1天(因此有些黑客认为这对于某些用例来说是个好主意)。较低的值还可能导致整页写入而导致写入放大。

假设我们决定用30分钟。

checkpoint_timeout = 30min

现在我们需要估计数据库在30分钟内将产生多少WAL，以便我们可以将其用于max_wal_size。有几种方法可以确定产生了多少WAL:

  • 使用pg_current_xlog_insert_location()查看实际的WAL位置(实质上是文件中的偏移量)，并计算每30分钟测量一次的位置之间的差异。

  • 启用log_checkpoint =on，然后从服务器日志中提取信息(将会有每个完成检查点的详细统计信息，包括WAL的数量)。

  • 使用来自pg_stat_bgwriter的数据，其中还包括关于检查点数量的信息(您可以结合当前max_wal_size值的知识)。

让我们以第一种方法为例。在我运行“pgbench”的测试机器上，我确实看到了这个:

postgres=# SELECT pg_current_xlog_insert_location();
 pg_current_xlog_insert_location 
---------------------------------
 3D/B4020A58
(1 row)

... after 5 minutes ...

postgres=# SELECT pg_current_xlog_insert_location();
 pg_current_xlog_insert_location 
---------------------------------
 3E/2203E0F8
(1 row)

postgres=# SELECT pg_xlog_location_diff('3E/2203E0F8', '3D/B4020A58');
 pg_xlog_location_diff 
-----------------------
            1845614240
(1 row)

这表明在5分钟内，数据库生成了约1.8GB的WAL，因此对于checkpoint_timeout = 30min，这将是大约10GB的WAL。然而，正如前面提到的，max_wal_size是2 - 3个检查点的组合配额，所以max_wal_size = 30GB (3 x 10GB)似乎是正确的。

其他方法使用不同的数据来源，但思想是相同的。

扩展检查点

我建议你需要调优checkpoint_timeout和max_wal_size，我没有告诉全部真相。还有另一个参数，叫做checkpoint_completion_target。但是要调优它，您需要理解“扩展检查点”的含义。

但是在CHECKPOINT期间，数据库需要执行以下三个基本步骤:

  1. 识别共享缓冲区中的所有脏(修改)块
  2. 将所有这些缓冲区写入磁盘(或者写入文件系统缓存)
  3. fsync()所有修改的文件到磁盘

只有当所有这些步骤都完成时，检查点才算完成。你可以“尽可能快地”完成这些步骤，即一次性写入所有脏缓冲区，然后在文件上调用fsync，事实上这就是PostgreSQL直到8.2所做的。但是这会导致I/O由于填充文件系统缓存和设备饱和而停滞，并影响用户会话。

为了解决这个问题，PostgreSQL 8.3引入了“扩展检查点”的概念，不是一次写入所有数据，而是在很长一段时间内进行写操作。这使得操作系统有时间在后台清除脏数据，使得最终的fsync更廉价。

写是基于到下一个检查点的进度来限制的，数据库知道在需要另一个“检查点”之前我们还剩下多少时间/ WAL，并且它可以计算出应该已经写出多少缓冲区。然而，数据库必须在最后才发出写操作，这意味着最后一批写操作仍然在文件系统缓存中，使得最后的fsync调用(在启动下一个检查点之前发出)再次变得昂贵。

因此，数据库需要给内核留出足够的时间，以便脏数据在后台被刷新到磁盘——而页面缓存(Linux文件系统缓存)的过期通常是由时间驱动的，特别是由内核参数：

vm.dirty_expire_centisecs = 3000

它表示数据在30秒后过期(默认情况下)。

注意: 当涉及到内核参数时，调优vm.dirty_background_bytes很重要。在具有大量内存的系统上，默认值太高，允许内核在文件系统缓存中积累大量脏数据。内核经常决定一次性刷新它们，减少了扩展检查点的好处。

现在，回到checkpoint_completion_target = 0.5。这个配置参数表示所有写操作应该完成到下一个检查点的距离。例如，假设检查点仅由checkpoint_timeout = 5min触发，数据库将限制写入，以便最后一次写入在2.5分钟后完成。然后，操作系统还有2.5分钟的时间将数据刷新到磁盘，因此5分钟后发出的fsync调用既便宜又快速。

当然，考虑到过期超时只有30秒，让系统停留2.5分钟似乎有些过分。你可以将checkpoint_completion_target增加到0.85，这样系统就会剩下45秒，比它需要的30秒多一点。但不建议这样做，因为在密集写入的情况下，检查点可能会被max_wal_size触发，而不是在5分钟之后，留给OS的时间少于30秒。

然而，处理写密集型工作负载的系统不太可能以更高的checkpoint_timeout值运行，这使得默认的completion_target值明显过低。例如，将超时设置为30分钟，它将强制数据库在前15分钟内完成所有写操作(以两倍的写速率)，然后在其余15分钟内无所事事。

相反，您可以尝试大致使用这个公式设置checkpoint_completion_target

(checkpoint_timeout - 2min) / checkpoint_timeout

30分钟大约是0.93。有时它建议不要超过0.9，这可能是可以的，并且您不太可能观察到这两个值之间的任何显著差异。(当使用非常高的checkpoint_timeout值时，这可能会改变，现在在PostgreSQL 9.6上可以达到1天)。

总结

现在您应该知道检查点的目的是什么，以及调优它们的基础知识。总结一下:

  • 大多数检查点应该是基于时间的，即由checkpoint_timeout

    • 性能(不频繁的检查点)和恢复所需时间(频繁的检查点)之间的妥协

    • 15-30分钟之间的值是最常见的，但上升到1小时也不是坏事

  • 在决定超时后，通过估计WAL的数量选择max_wal_size

  • 设置checkpoint_completion_target，以便内核有足够的时间(但不是不必要的太多)将数据刷新到磁盘

  • 也调优vm.Dirty_background_bytes来防止内核在页面缓存中积累大量脏数据

或者您可以简单地使用我们的性能调优服务，该服务不仅包括检查点，还包括所有其他重要的配置部分。
