EXPLAIN – 第1部分

postgresql
数据库
发布日期:   2023-11-02

本篇目标是帮助您完成理解和优化查询以提高性能的完整过程。PostgreSQL 服务器尝试找到构建查询结果集的最有效方法。朝这个方向迈出的第一步是能够理解命令EXPLAIN是什么。

当一条语句被发送到 PostgreSQL 服务器执行时，它必须破译该查询的各个部分并定义一个执行计划。鉴于可以通过各种不同的方式检索和管理/操作数据，PostgreSQL 服务器尝试找到构建查询结果集的最有效方法。特别是在性能很重要的长时间运行查询的情况下，在给定平台上使用现有服务器配置执行语句的执行计划决定了整体查询（在大多数情况下是应用程序）性能。因此，了解 PostgreSQL 服务器如何以执行计划的形式分解查询执行非常重要。这就是EXPLAIN命令的用武之地。用非常基本的术语来说，EXPLAIN 命令显示（解释）语句的执行计划。

PostgreSQL 中的查询处理分为 5 个主要组件：

  1. 解析器
  2. 分析器
  3. 重写器
  4. 规划器
  5. 执行器

而 (1) 解析器和 (2) 分析器负责确保正确编写查询，而结果是一个查询树，该查询树被传递到 (3) 重写器，重写器可以根据 pg_rules 系统目录表中定义的规则对其进行转换。因此，这个转换后的查询树被传递到 (4) 规划器，该规划器生成由 (5) 执行器执行的计划。

解释命令

“EXPLAIN”命令具有PostgreSQL 文档中定义的简单语法。EXPLAIN命令仅解释给定的语句，但是当使用该ANALYZE选项运行时，它也会执行该语句。结果集永远不会返回给客户端。

因此请注意，运行会EXPLAIN ANALYZE解释并运行给定的语句。让我们看看EXPLAIN命令的输出，首先创建并填充一个名为的表foo_explain，然后在foo_explain上的 select 语句上运行EXPLAIN命令。

postgres=# CREATE TABLE foo_explain(A INT);
CREATE TABLE
 
postgres=# INSERT INTO foo_explain SELECT GENERATE_SERIES(1,50000);
INSERT 0 50000

postgres=# EXPLAIN SELECT * FROM foo_explain;
                                    QUERY PLAN                                    
----------------------------------------------------------------------------------
 Seq Scan on foo_explain  (cost=10000000000.00..10000000015.00 rows=50000 width=4)
(1 row)

EXPLAIN 命令的输出

生成的查询计划是一个非常基本的计划，但是我们需要了解一些重要的部分才能理解计划器所做的事情。

查询计划告诉我们行获取策略、表foo_explain上的Seq Scan（顺序扫描）、该操作的启动成本10000000000.00以及Seq Scan的总成本10000000015.00。它还给出了预期返回的行数 (1000) 以及行的大小 (4 字节) 的估计值。

启动成本和总成本是任意数字，为规划者提供了一种比较不同策略以选择最佳策略的方法。一般来说，成本越低表明执行速度越快。

错误的估计

到目前为止，一切都很好？了解行和宽度是估计值而不是实际数字非常重要。因此，这些有时可能不准确。这是我的意思的一个例子。

postgres=# CREATE TABLE t(a INT, b VARCHAR(1));
CREATE TABLE
postgres=# INSERT INTO t VALUES(1, 'h');
INSERT 0 1
postgres=# SELECT * FROM t;
 a |   b    
---+--------
 1 | hello1
(1 row)
 
postgres=# EXPLAIN SELECT * FROM t;
                               QUERY PLAN                                
-------------------------------------------------------------------------
 Seq Scan on t  (cost=10000000000.00..10000000030.40 rows=2040 width=12)
(1 row)

哎哟! 由于一些非常奇怪的原因，规划器认为我们的 select 语句将返回 2040 行，每行 12 个字节，如果我们将这两个数字相乘，则总计 24,480 个字节。那么这里出了什么问题呢？行和宽度字段是基于 PostgreSQL 的一些表数据和统计数据的最佳猜测。这些可以在pg_class和pg_stats表中轻松看到。

postgres=# SELECT relname, relkind, reltuples, relpages FROM pg_class WHERE relname = 't';
 relname | relkind | reltuples | relpages 
---------+---------+-----------+----------
 t       | r       |         0 |        0
(1 row)
 
postgres=# SELECT attname, inherited, n_distinct,
postgres-#        array_to_string(most_common_vals, E'\n') as most_common_vals
postgres-# FROM pg_stats
postgres-# WHERE tablename = 't';
 attname | inherited | n_distinct | most_common_vals 
---------+-----------+------------+------------------
(0 rows)

很明显，我们的服务器认为我们的表中没有元组，因此也没有relpages，也没有统计数据来帮助我们更好地估计 select 语句的输出。通过在表上运行VACUUM ANALYZE命令可以轻松解决此问题。

postgres=# VACUUM ANALYZE t;
VACUUM

重新查询pg_class现在pg_stats给我们一个更清晰的画面。

postgres=# SELECT relname, relkind, reltuples, relpages FROM pg_class WHERE relname = 't';
 relname | relkind | reltuples | relpages 
---------+---------+-----------+----------
 t       | r       |         1 |        1
(1 row)
 
postgres=# SELECT attname, inherited, n_distinct,
postgres-#        array_to_string(most_common_vals, E'\n') as most_common_vals
postgres-# FROM pg_stats
postgres-# WHERE tablename = 't';
 attname | inherited | n_distinct | most_common_vals 
---------+-----------+------------+------------------
 a       | f         |         -1 | 
 b       | f         |         -1 | 
(2 rows)

这个输出更有意义。让我们再次运行EXPLAIN。

postgres=# EXPLAIN SELECT * FROM t;
                             QUERY PLAN                              
---------------------------------------------------------------------
 Seq Scan on t  (cost=10000000000.00..10000000001.01 rows=1 width=6)
(1 row)

现在显然这是表格的实际反映。正如预期的那样，只会返回 1 行，并且行的大小为 6 字节，而之前给出的为 12 字节。不用说，糟糕的统计数据可能会导致计划不理想，从而导致查询性能不佳。

根据给定计划输出构建查询

如果我们可以对给定的计划输出进行逆向工程并构建查询，那么我们可以有把握地说我们理解了输出。鉴于以下计划不是最复杂的计划，它仍然具有各种不同的节点类型。为了简单起见，使用VERBOSE ON选项运行以下EXPLAIN查询。

                                           QUERY PLAN                                           
------------------------------------------------------------------------------------------------
 GroupAggregate  (cost=10000000010.30..10000000010.35 rows=1 width=23)
   Output: count(*), t1.id, t1.name, t2.id
   Group Key: t1.id, t1.name, t2.id
   Filter: (count(*) > 1)
   ->  Sort  (cost=10000000010.30..10000000010.30 rows=2 width=15)
         Output: t1.id, t1.name, t2.id
         Sort Key: t1.id DESC, t1.name
         ->  Nested Loop  (cost=10000000004.20..10000000010.29 rows=2 width=15)
               Output: t1.id, t1.name, t2.id
               Join Filter: (t1.id = t2.id)
               ->  Seq Scan on public.t1  (cost=10000000000.00..10000000001.25 rows=6 width=11)
                     Output: t1.id, t1.name
                     Filter: (t1.id < 8)
               ->  Materialize  (cost=4.20..8.34 rows=8 width=4)
                     Output: t2.id
                     ->  Bitmap Heap Scan on public.t2  (cost=4.20..8.30 rows=8 width=4)
                           Output: t2.id
                           Recheck Cond: (t2.id < 5)
                           ->  Bitmap Index Scan on t2_idx_id  (cost=0.00..4.20 rows=8 width=0)
                                 Index Cond: (t2.id < 5)
(20 rows)

在我们开始更详细地阅读该计划之前，我们看到第 4 行指示了选定的列。所以我们的 select 语句以 开头SELECT COUNT(*), t1.id, t1.name, t2.id。

现在更进一步，从叶节点开始向上遍历，重新设计 SQL 查询：

  • 第20行展示了对 t2.id 的过滤条件；即可能是 WHERE 子句的一部分；t2.id < 5
  • 第 16 – 22 行表明表 t2 是我们查询的一部分。此外，这些还表明计划正在使用表 t2 上的索引 t2_idx_id。
  • 同样，第 13 – 15 行告诉我们表 t1 是我们查询的一部分，并且具有过滤条件t1.id < 8。
  • 第 10 – 12 行告诉我们它是一个INNER JOIN ON t1.id = t2.id.

因此，计划的这些部分表明 where 子句和 from 子句以及我们目前的查询如下所示：

SELECT COUNT(*), t1.id, t1.name, t2.id 
FROM t1 INNER JOIN t2
ON t1.id = t2.id
WHERE t1.id < 8 AND t2.id < 5
GROUP BY t1.id, t1.name, t2.id
HAVING COUNT(*) > 1
ORDER BY t1.id DESC, t1.name;

这与生成查询计划的原始查询相匹配。

结论

在本博客中，我们了解了该EXPLAIN命令的基础知识，为什么确保统计信息是最新的以实现良好的查询性能非常重要。在下一篇博客中，我将更深入地研究EXPLAIN命令的输出，以及稍微复杂的查询，以及索引、扫描、联接和排序方法等不同因素会影响查询计划。敬请关注！
