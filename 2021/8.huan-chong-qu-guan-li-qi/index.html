8 缓冲区管理器

postgresql
postgresql
发布日期:   2021-08-13

缓冲区管理器管理共享内存和持久存储之间的数据传输，并且可以对 DBMS 的性能产生重大影响。PostgreSQL 缓冲区管理器非常有效地工作。

在本章中，描述了 PostgreSQL 缓冲区管理器。第一部分提供概述，随后的部分描述以下主题：

  • 缓冲区管理器结构
  • 缓冲区管理器锁
  • 缓冲区管理器的工作原理
  • 环形缓冲区
  • 刷新脏页

8.1. 概述

本节介绍了为方便后续章节的描述所需的关键概念。

8.1.1 缓冲区管理器结构

PostgreSQL 缓冲管理器包括缓冲表、缓冲描述符和缓冲池，它们将在下一节中描述。该缓冲池层存储数据文件页面，如表和索引，以及自由空间地图和可见性地图。缓冲池是一个数组，即每个槽存储一个数据文件的一页。缓冲池数组的索引称为buffer_id。

8.2和8.3节描述了缓冲区管理器内部的细节。

8.1.2 缓冲标签

在 PostgreSQL 中，所有数据文件的每一页都可以分配一个唯一的标签，即缓冲区标签。当缓冲区管理器收到请求时，PostgreSQL 使用所需页面的 buffer_tag。

所述buffer_tag包括三个值：RelFileNode和到它的页面所属的关系的叉号，和它的页的块号。表、自由空间映射和可见性映射的分叉数分别定义为 0、1 和 2。

例如，buffer_tag ‘{(16821, 16384, 37721), 0, 7}’ 标识关系的 OID 和分叉号分别为 37721 和 0 的第七个块中的页面；该关系包含在 OID 为 16821 的表空间下的 OID 为 16384 的数据库中。 类似地，buffer_tag ‘{(16821, 16384, 37721), 1, 3}’ 标识位于空闲空间第三块中的页OID 和叉号分别为 37721 和 1 的地图。

8.1.3 后端进程如何读取页面

本小节描述后端进程如何从缓冲区管理器读取页面（图 8.2）。

  • (1) 读取表或索引页时，后端进程向缓冲区管理器发送包含该页的buffer_tag 的请求。
  • (2) 缓冲区管理器返回存储请求页面的槽的buffer_ID。如果请求的页面未存储在缓冲池中，则缓冲管理器将页面从持久存储加载到缓冲池槽之一，然后返回 buffer_ID 的槽。
  • (3) 后端进程访问buffer_ID的slot（读取需要的页面）。

当后端进程修改缓冲池中的页面（例如，通过插入元组）时，尚未刷新到存储的已修改页面被称为脏页面。

8.4 节描述了缓冲区管理器的工作原理。

8.1.4 页替换算法

当所有缓冲池槽位都被占用但请求的页面没有被存储时，缓冲管理器必须在缓冲池中选择一个将被请求的页面替换的页面。通常，在计算机科学领域，页面选择算法被称为页面替换算法 ，被选中的页面被称为受害者页面。

自从计算机科学出现以来，页面替换算法的研究就一直在进行。因此，之前已经提出了许多替换算法。从 8.1 版本开始，PostgreSQL 开始使用时钟扫描，因为它比以前版本中使用的 LRU 算法更简单、更高效。

8.4.4 节描述了时钟扫描的细节。

8.1.5 冲洗脏页面

脏页最终应该被刷新到存储中；但是，缓冲区管理器需要帮助才能执行此任务。在 PostgreSQL 中，有两个后台进程checkpointer和background writer负责这个任务。

8.6 节描述了检查指针和后台编写器。

直接输入/输出

PostgreSQL里面不支持直接I / O，尽管有时它已被讨论。如果您想了解更多详细信息，请参阅有关 pgsql-ML 的讨论和这篇文章。

8.2 缓冲区管理器结构

PostgreSQL 缓冲管理器包括三层，即缓冲表、缓冲描述符和缓冲池（图 8.3）：

  • 所述缓冲器池是一个数组。每个插槽存储一个数据文件页。数组槽的索引被称为buffer_id s。
  • 该缓冲区描述符层是缓冲区描述符的数组。每个描述符与缓冲池槽一一对应，并在相应槽中保存存储页面的元数据。
    请注意，为方便起见采用了术语“缓冲区描述符层”，它仅在本文档中使用。
  • 该缓存表是一个哈希表存储之间的关系buffer_tag存网页的S和buffer_id持有的存网页各自的元数据描述符秒。

这些层将在以下小节中详细描述。

8.2.1 缓冲表

缓冲表在逻辑上可以分为三部分：散列函数、散列桶槽和数据条目（图 8.4）。

内置的哈希函数将 buffer_tags 映射到哈希桶槽。即使哈希桶槽数大于缓冲池槽数，也可能发生冲突。因此，缓冲表使用单独的链表链接方法来解决冲突。当数据条目映射到同一个桶槽时，该方法将条目存储在同一个链表中，如图8.4所示。

数据条目包含两个值：页面的 buffer_tag 和保存页面元数据的描述符的 buffer_id。例如，数据条目“ Tag_A, id=1 ”表示buffer_id 为1的缓冲区描述符存储用Tag_A标记的页面的元数据。

哈希函数

哈希函数是calc_bucket()和 hash()的复合函数。 以下是它作为伪函数的表示。

uint32 bucket_slot = calc_bucket(unsigned hash(BufferTag buffer_tag), uint32 bucket_size)

请注意，基本操作（数据条目的查找、插入和删除）在此不做说明。这些是非常常见的操作，将在以下部分进行说明。

8.2.2 缓冲区描述符

本节介绍缓冲区描述符的结构，下一节介绍缓冲区描述符层。

缓冲区描述符将存储页面的元数据保存在相应的缓冲池槽中。缓冲区描述符结构由结构BufferDesc定义。虽然这个结构体有很多字段，但主要如下所示：

  • tag在相应的缓冲池槽中保存存储页面的buffer_tag（缓冲区标签在第 8.1.2 节中定义 ）。

  • buffer_id标识描述符（相当于对应缓冲池槽的buffer_id）。

  • refcount保存当前访问相关存储页面的 PostgreSQL 进程数。它也称为引脚数。当 PostgreSQL 进程访问存储的页面时，它的 refcount 必须加 1（refcount++）。访问页面后，它的refcount必须减1（refcount–）。
    当引用计数为零时，即当前没有访问关联的存储页，则该页被取消固定；否则它被固定。

  • usage_count保存关联存储页面自加载到相应缓冲池槽以来被访问的次数。请注意，usage_count 用于页面替换算法（第 8.4.4 节）。

  • context_lock和io_in_progress_lock是轻量级锁，用于控制对相关存储页面的访问。这些字段在第 8.3.2 节中描述。

  • flags

    可以保存相关存储页面的多个状态。主要状态如下：

    • 脏位指示存储的页面是否脏。
    • 有效位指示存储的页面是否可以读取或写入（有效）。例如，如果这个位有效，那么对应的缓冲池槽存储一个页面，这个描述符（有效位）保存页面元数据；因此，可以读取或写入存储的页面。如果此位无效，则此描述符不包含任何元数据；这意味着无法读取或写入存储的页面，或者缓冲区管理器正在替换存储的页面。
    • io_in_progress 位指示缓冲区管理器是否正在从/向存储读取/写入关联的页面。换句话说，该位指示是否有单个进程持有该描述符的 io_in_progress_lock。

  • freeNext是指向下一个描述符以生成freelist的指针，这将在下一小节中描述。

结构BufferDesc在src/include/storage/buf_internals.h 中定义。

为了简化以下描述，定义了三个描述符状态：

  • 空：当相应的缓冲池槽不存储页面（即引用计数和usage_count是0），该描述符的状态是空的。
  • 钉扎：当相应的缓冲池时隙存储页面和任何的PostgreSQL进程正在访问的页面（即，引用计数和usage_count是大于或等于1），该缓冲区描述符的状态被固定。
  • Unpinned：当对应的缓冲池槽存储了一个页面，但没有PostgreSQL进程访问该页面时（即usage_count大于等于1，但refcount为0），这个缓冲区描述符的状态为unpinned。

每个描述符将具有上述状态之一。描述符状态相对于特定条件而变化，这将在下一小节中描述。

在下图中，缓冲区描述符的状态由彩色框表示。

  • (白色)空
  • (蓝色)固定
  • （水蓝色）未固定

此外，脏页用“X”表示。例如，未固定的脏描述符由 X 表示。

8.2.3 缓冲区描述符层

缓冲区描述符的集合形成一个数组。在本文档中，该数组被称为缓冲区描述符层。

当 PostgreSQL 服务器启动时，所有缓冲区描述符的状态都是empty。在 PostgreSQL 中，这些描述符包含一个名为freelist的链表（图 8.5）。

请注意，空闲列表在PostgreSQL是从完全不同的概念空闲列表中的Oracle。PostgreSQL 的 freelist 只是空缓冲区描述符的链表。在5.3.4 节中描述的PostgreSQL自由空间映射中，充当与 Oracle 中的空闲列表相同的角色。

图 8.6 显示了如何加载第一页。

  • (1) 从freelist的顶部取回一个空的descriptor，并将其固定（即把它的refcount和usage_count加1）。
  • (2) 在缓冲区表中插入新条目，该条目保存第一页的标签和检索到的描述符的buffer_id 之间的关系。
  • (3) 将新页面从存储加载到相应的缓冲池槽中。
  • (4) 将新页面的元数据保存到检索到的描述符中。

第二页和后续页的加载方式类似。第 8.4.2 节提供了其他详细信息。

从空闲列表中检索到的描述符始终保存页面的元数据。换句话说，非空描述符继续使用不会返回到freelist。但是，当发生以下任一情况时，相关描述符会再次添加到空闲列表中，并且描述符状态变为“空”：

  1. 表或索引已被删除。
  2. 数据库已被删除。
  3. 已使用 VACUUM FULL 命令清理表或索引。

为什么空描述符包含空闲列表？

做freelist的原因是为了立即获取第一个描述符。这是动态内存资源分配的常见做法。请参阅此说明。

缓冲区描述符层包含一个无符号的 32 位整数变量，即nextVictimBuffer。该变量用于第 8.4.4 节中描述的页面替换算法。

8.2.4 缓冲池

缓冲池是一个简单的数组，用于存储数据文件页，例如表和索引。缓冲池数组的索引被称为buffer_id s。

缓冲池槽大小为 8 KB，等于一个页面的大小。因此，每个槽可以存储一整页。

8.3 缓冲区管理器锁

缓冲区管理器将许多锁用于许多不同的目的。本节描述了后续章节中解释所需的锁。

请注意，本节中描述的锁是缓冲区管理器同步机制的一部分；他们根本不涉及到任何SQL语句和SQL选项。

8.3.1 缓冲表锁

BufMappingLock保护整个缓冲表的数据完整性。它是一种轻量级锁，可用于共享和独占模式。在缓冲区表中搜索条目时，后端进程持有共享的 BufMappingLock。插入或删除条目时，后端进程持有排他锁。

BufMappingLock 被拆分成多个分区以减少缓冲表中的争用（默认为 128 个分区）。每个 BufMappingLock 分区保护相应哈希桶槽的部分。

图 8.7 是一个典型的 BufMappingLock 拆分效果示例。两个后端进程可以同时以独占模式持有各自的 BufMappingLock 分区，以便插入新的数据条目。如果 BufMappingLock 是单个系统范围的锁，则两个进程都应等待另一个进程的处理，具体取决于哪个进程开始处理。

缓冲表需要许多其他锁。例如，缓冲表内部使用自旋锁来删除条目。然而，这些其他锁的描述被省略，因为它们在本文档中不是必需的。

在 9.4 版本之前，BufMappingLock 默认被拆分为 16 个单独的锁。

8.3.2 每个缓冲区描述符的锁

每个缓冲区描述符使用两个轻量级锁content_lock和io_in_progress_lock来控制对相应缓冲池槽中存储页面的访问。当检查或更改自己字段的值时，将使用自旋锁。

8.3.2.1 内容锁

content_lock 是强制访问限制的典型锁。它可以在共享和独占模式下使用。

在读取页面时，后端进程会获取存储该页面的缓冲区描述符的共享 content_lock。

但是，在执行以下操作之一时会获得一个独占的 content_lock：

  • 将行（即元组）插入存储页或更改存储页内元组的 t_xmin/t_xmax 字段（t_xmin 和 t_xmax 在第 5.2 节中描述；简单地说，当删除或更新行时，关联元组的这些字段被更改） .
  • 除去元组物理或所存储的页上的压实自由空间（通过真空处理和HOT，其中所述进行章节6和7，分别地）。
  • 冻结存储页面中的元组（冻结在第 5.10.1节和第 6.3节中描述）。

官方README 文件显示了更多详细信息。

8.3.2.2 io_in_progress_lock

io_in_progress 锁用于等待缓冲区上的 I/O 完成。当 PostgreSQL 进程从/向存储加载/写入页面数据时，该进程在访问存储时持有相应描述符的独占 io_in_progress 锁。

8.3.2.3 自旋锁

当检查或更改标志或其他字段（例如 refcount 和 usage_count）时，将使用自旋锁。下面给出了两个自旋锁使用的具体例子：

  • (1) 下面显示了如何固定缓冲区描述符：

    • \1. 获取缓冲区描述符的自旋锁。
    • 2.将其refcount和usage_count的值加1。
    • \3. 释放自旋锁。
  • LockBufHdr(bufferdesc);    /* Acquire a spinlock */
bufferdesc->refcont++;
bufferdesc->usage_count++;
UnlockBufHdr(bufferdesc); /* Release the spinlock */

  • (2) 下面显示了如何将脏位设置为“1”：

    • \1. 获取缓冲区描述符的自旋锁。
    • \2. 使用按位运算将脏位设置为“1”。
    • \3. 释放自旋锁。
  • #define BM_DIRTY             (1 << 0)    /* data needs writing */
#define BM_VALID             (1 << 1)    /* data is valid */
#define BM_TAG_VALID         (1 << 2)    /* tag is assigned */
#define BM_IO_IN_PROGRESS    (1 << 3)    /* read or write in progress */
#define BM_JUST_DIRTIED      (1 << 5)    /* dirtied since write started */

LockBufHdr(bufferdesc);
bufferdesc->flags |= BM_DIRTY;
UnlockBufHdr(bufferdesc);

  • 以相同的方式更改其他位。

用原子操作替换缓冲区管理器自旋锁

在 9.6 版本中，缓冲区管理器的自旋锁将被替换为原子操作。查看commitfest 的这个结果。如果您想了解详细信息，请参阅此讨论。

8.4 缓冲区管理器的工作原理

本节介绍缓冲区管理器的工作原理。当后端进程想要访问所需的页面时，它会调用ReadBufferExtended函数。

ReadBufferExtended函数 的行为取决于三种逻辑情况。以下小节对每种情况进行了描述。此外，PostgreSQL时钟扫描页面替换算法在最后一小节中描述。

8.4.1 访问存储在缓冲池中的页面

首先，描述最简单的情况，即所需页面已经存储在缓冲池中。在这种情况下，缓冲区管理器执行以下步骤：

  • (1) 创建所需页面的buffer_tag（在本例中，buffer_tag 为’Tag_C’）并使用哈希函数计算哈希桶槽，其中包含创建的buffer_tag的关联条目。
  • (2) 在共享模式下获取覆盖获取的hash桶槽的BufMappingLock分区（这个锁会在步骤（5）中释放）。
  • (3) 查找tag为’Tag_C’的表项，从表项中获取buffer_id。在此示例中，buffer_id 为 2。
  • (4) pin the buffer descriptor for buffer_id 2，即描述符的refcount和usage_count加1（8.3.2节描述pinning）。
  • (5) 释放BufMappingLock。
  • (6) 访问buffer_id为2的缓冲池槽。

然后，当从缓冲池槽中的页面读取行时，PostgreSQL 进程获取相应缓冲描述符的共享 content_lock。因此，多个进程可以同时读取缓冲池槽。

当向页面插入（和更新或删除）行时，Postgres 进程获取相应缓冲区描述符的独占 content_lock（注意页面的脏位必须设置为“1”）。

访问页面后，相应缓冲区描述符的 refcount 值减 1。

8.4.2 将页面从存储加载到空槽

在第二种情况下，假设所需的页面不在缓冲池中，并且空闲列表具有空闲元素（空描述符）。在这种情况下，缓冲区管理器执行以下步骤：

  • (1) 查找缓冲表（我们假设没有找到）。
    • \1. 创建所需页面的buffer_tag（在本例中，buffer_tag 为’Tag_E’）并计算哈希桶槽。
    • \2. 获取共享模式下的BufMappingLock分区。
    • 3.查找缓冲表（根据假设没有找到）。
    • \4. 释放 BufMappingLock。
  • (2)从freelist中获取空缓冲区描述符，并固定。本例中，获取到的描述符的buffer_id为4。
  • (3) 以独占方式获取BufMappingLock分区（该锁将在步骤（6）中释放）。
  • (4) 创建一个包含buffer_tag ‘Tag_E’和buffer_id 4的新数据条目；将创建的条目插入缓冲表。
  • (5) 将需要的页面数据从storage加载到buffer_id为4的缓冲池槽中，如下：
    • 1、获取对应描述符的独占io_in_progress_lock。
    • \2. 将相应描述符的io_in_progress位设置为’1，以防止其他进程访问。
    • \3. 将所需的页面数据从存储加载到缓冲池槽。
    • 4、改变对应描述符的状态；所述io_in_progress位被设置为“0”，并且有效位被设置为“1”。
    • \5. 释放io_in_progress_lock。
  • (6) 释放BufMappingLock。
  • (7) 访问buffer_id为4的缓冲池槽。

8.4.3 将页面从存储加载到受害者缓冲池槽

在这种情况下，假设所有缓冲池槽都被页面占用，但没有存储所需的页面。缓冲区管理器执行以下步骤：

  • (1)创建所需页面的buffer_tag，查找缓冲表。在此示例中，我们假设 buffer_tag 是“Tag_M”（未找到所需页面）。

  • (2) 使用时钟扫描算法选择一个受害者缓冲池槽，从缓冲表中获取包含受害者池槽的buffer_id的旧条目，并将受害者池槽固定在缓冲区描述符层。在这个例子中，受害者槽的 buffer_id 是 5，旧条目是 ‘Tag_F, id=5’。时钟扫描将在下一小节中描述。

  • (3) 如果受害页面数据是脏的，则刷新（写入和 fsync）；否则继续步骤（4）。

    在用新数据覆盖之前，必须将脏页写入存储。刷新脏页执行如下：

    • 1、获取buffer_id为5的描述符的共享content_lock和独占io_in_progress锁（步骤6中释放）。
    • 2.改变对应描述符的状态；所述io_in_progress位被设置为“1”和just_dirtied位被设置为“0”。
    • 3、根据情况，调用XLogFlush()函数将WAL缓冲区上的WAL数据写入当前的WAL段文件中（细节省略；WAL和XLogFlush函数在第9章描述）。
    • \4. 将受害者页面数据刷新到存储中。
    • 5、改变对应描述符的状态；所述io_in_progress位被设置为“0”和有效位被设置为“1”。
    • \6. 释放 io_in_progress 和 content_lock 锁。

  • (4) 以独占模式获取覆盖包含旧条目的槽的旧BufMappingLock分区。

  • (5) 获取新的BufMappingLock分区，将新条目插入缓冲表：

    • \1. 创建由新的 buffer_tag ‘Tag_M’ 和受害者的 buffer_id 组成的新条目。
    • \2. 以独占模式获取新的 BufMappingLock 分区，该分区覆盖包含新条目的槽。
    • \3. 将新条目插入缓冲表。

  • (6) 从缓冲表中删除旧条目，并释放旧的 BufMappingLock 分区。
  • (7) 将所需的页面数据从存储加载到受害缓冲区插槽。然后，使用 buffer_id 5 更新描述符的标志；脏位设置为“0”并初始化其他位。
  • (8) 释放新的BufMappingLock分区。
  • (9) 访问buffer_id为5的缓冲池槽。

8.4.4 页替换算法：时钟扫描

本节的其余部分描述时钟扫描算法。该算法是NFU（不常用）的一种变体，开销较低；它有效地选择不太常用的页面。

将缓冲区描述符想象成一个循环列表（图 8.12）。nextVictimBuffer 是一个无符号的 32 位整数，始终指向缓冲区描述符之一并顺时针旋转。算法的伪代码和描述如下：

伪代码：时钟扫描

     WHILE true
(1)     Obtain the candidate buffer descriptor pointed by the nextVictimBuffer
(2)     IF the candidate descriptor is unpinned THEN
(3)	       IF the candidate descriptor's usage_count == 0 THEN
                BREAK WHILE LOOP  /* the corresponding slot of this descriptor is victim slot. */
           ELSE
            Decrease the candidate descriptpor's usage_count by 1
               END IF
         END IF
(4)     Advance nextVictimBuffer to the next one
      END WHILE 
(5) RETURN buffer_id of the victim
  • (1) 获取nextVictimBuffer指向的候选缓冲区描述符。
  • (2) 如果候选缓冲区描述符是unpinned，则进行步骤(3)；否则，继续执行步骤（4）。
  • (3)如果候选描述符的usage_count为0，则选择该描述符对应的slot作为牺牲品，进入步骤(5)；否则，将此描述符的usage_count减少 1 并继续执行步骤 (4)。
  • (4) 将 nextVictimBuffer 推进到下一个描述符（如果在最后，则回绕）并返回到步骤（1）。重复直到找到受害者。
  • (5) 返回受害者的buffer_id。

具体示例如图 8.12 所示。缓冲区描述符显示为蓝色或青色框，框中的数字表示每个描述符的使用次数。

  • 1）nextVictimBuffer指向第一个描述符（buffer_id 1）；然而，这个描述符被跳过，因为它是固定的。
  • \2) nextVictimBuffer 指向第二个描述符（buffer_id 2）。这个描述符是 unpinned 但它的 usage_count 是 2；因此，usage_count 减 1，nextVictimBuffer 前进到第三个候选者。
  • 3）nextVictimBuffer指向第三个描述符（buffer_id 3）。这个描述符是 unpinned，它的 usage_count 是 0；因此，这是本轮的受害者。

每当nextVictimBuffer扫描一个unpinned描述符时，它的usage_count就会减 1。因此，如果缓冲池中存在unpinned描述符，这个算法总是可以通过旋转 nextVictimBuffer 来找到一个 usage_count 为 0 的受害者。

8.5 环形缓冲器

在读取或写入大表时，PostgreSQL 使用环形缓冲区而不是缓冲池。该环形缓冲器是一个小而临时缓冲区域。当满足下面列出的任何条件时，都会为共享内存分配一个环形缓冲区：

  1. 批量阅读

  2. 扫描大小超过缓冲池大小 (shared_buffers/4) 四分之一的关系时。在这种情况下，环形缓冲区大小为256 KB。

  3. 批量写入

  4. 当下面列出的 SQL 命令被执行时。在这种情况下，环形缓冲区大小为16 MB。

    • *从命令复制*。
    • *CREATE TABLE AS*命令。
    • CREATE MATERIALIZED VIEW或 REFRESH MATERIALIZED VIEW命令。
    • 更改表命令。

  6. 真空处理

  7. 当自动真空吸尘器执行真空处理时。在这种情况下，环形缓冲区大小为

  8. 256 KB

  9. 分配的环形缓冲区在使用后立即释放。环形缓冲区的好处是显而易见的。如果后端进程在不使用环形缓冲区的情况下读取大表，则将缓冲池中所有存储的页面删除（踢出）；因此，缓存命中率降低。环形缓冲区避免了这个问题。

为什么批量读取和真空处理的默认环形缓冲区大小是 256 KB？

为什么是 256 KB？答案在位于缓冲区管理器源目录下的自述文件中进行了解释。

对于顺序扫描，使用 256 KB 环。它小到足以放入 L2 缓存，这使得将页面从 OS 缓存传输到共享缓冲区缓存变得高效。甚至更少通常就足够了，但环必须足够大以容纳扫描中同时固定的所有页面。（剪辑）

8.6 冲洗脏页面

除了替换受害页面之外，检查指针和后台编写器进程将脏页面刷新到存储中。两个进程具有相同的功能（刷新脏页）；然而，他们有不同的角色和行为。

检查点进程将检查点记录写入 WAL 段文件，并在检查点开始时刷新脏页。第 9.7 节描述了检查点及其开始时间。

后台写入器的作用是减少checkpointing的密集写入的影响。后台编写器继续一点一点地刷新脏页，对数据库活动的影响最小。默认情况下，后台编写器每 200 毫秒唤醒一次（由bgwriter_delay定义）并最多刷新bgwriter_lru_maxpages（默认为 100 页）。

为什么检查指针与后台编写器分开？

在 9.1 或更早版本中，后台编写器会定期进行检查点处理。在 9.2 版本中，检查点进程已与后台写入进程分离。由于标题为“分离bgwriter和checkpointer”的提案中描述了原因，因此其语句如下所示。
