TOAST介绍

postgresql
postgresql
发布日期:   2021-08-13

背景

TOAST是“The Oversized-Attribute Storage Technique”的缩写，主要用于存储一个大字段的值。由于Postgresql的页面大小是固定的（通常是8KB），并且不允许行跨越多个页面，因此不可能直接存储非常大的字段值。为了突破这个限制，大的字段值通常被压缩或切片成多个物理行存储到另一张系统表中（TOAST）。只有特定的数据类型支持TOAST，这也很好理解，那些整数、浮点等不太长的数据类型是没有必要使用TOAST的。另外，支持TOAST的数据类型必须是变长的。在变长类型中，前4字节（32bit）称为长度字，长度字后面存储具体的内容或一个指针。长度字的高2bit位是标志位，后面的30bit是长度值，长度值中包含了自身占用的这4个字节。由此可见TOAST数据类型的逻辑长度最多是30bit，即1GB（2^30-1）字节之内。前2bit的标志位，一个表示压缩标志位，一个表示是否行外存储，如果两个都是0，那么表示既未压缩也未行外存储。如果设置了第一个位，那么表示该数据被压缩过（使用的是非常简单且快速的LZ压缩方法），使用前必须先解压缩。如果设置了另外一个位，则表示该数值是在行外存储的。这个时候，长度后面的部分只是一个指针，指向存储实际数据的TOAST表的位置。如果两个位置都设置了，那么这个行外数据也会被压缩。不管是哪种情况，长度字里剩下的30bit的长度值都表示数据的实际尺寸，而不是压缩后的长度。

TOAST表不能独立创建，只有当普通表包含了main,extended或external存储格式的字段时，系统会自动创建一个和普通表关联的TOAST表。当一条记录(tuple)在存储时(如果压缩的话算压缩后的大小)大于TOAST_TUPLE_THRESHOLD(通常是2kB)这个值时，会存储到TOAST表。而此时在普通表的该字段处包含了一个指向TOAST的tableoid和chunk_id的数据，从而能够找到该字段的记录。

每个字段有四种TOAST的策略。对于定长的字段类型，存储格式如下：

  • PLAIN：避免压缩会行外存储。对于那些不能TOAST的数据类型可选择这个策略。

对于变长的字段类型，除了可以使用PLAIN格式，还可以使用如下存储格式：

  • EXTENDED：允许压缩和行外存储。这是大多数可以TOAST的数据类型的默认策略。首先进行压缩，如果行依然增大，则进行行外存储。

  • EXTERNA：允许行外存储，但是不允许压缩。因为没有压缩，这让在text类型和bytea类型字段上的子字符串操作更快，原因是这些子字符串操作可以只读整个数据中需要的部分，而不是整个数据，当然代价是增加了存储空间。

  • MAIN：允许压缩，但不允许行外存储。实际上，当行类无法放下时任然会进行行外存储，但只是在没有办法把数据行变得更小的情况下的最后手段。

例如我们来创建一个测试表：
根据前面的介绍，TOAST表只在有变长字段，并且存储为main,extended或external时才会创建。

jinli=# create table test_toast1(id int);
CREATE TABLE
jinli=# select relname,reltoastrelid from pg_class where relname=’test_toast1’;
relname | reltoastrelid
————-+—————
test_toast1 | 0
(1 row)

jinli=# \d+ test_toast1
Table “public.test_toast1”
Column | Type | Modifiers | Storage | Stats target | Description
——–+———+———–+———+————–+————-
id | integer | | plain | |
Has OIDs: no

原因很简单，定长类型以PLAIN存储。
如下例，创建了test_toast2表，系统字段创建了一个OID=49422的TOAST表。

jinli=# create table test_toast2(id int,author name,title varchar(256),content text);
CREATE TABLE
jinli=# \d+ test_toast2
Table “public.test_toast2”
Column | Type | Modifiers | Storage | Stats target | Description
———+————————+———–+———-+————–+————-
id | integer | | plain | |
author | name | | plain | |
title | character varying(256) | | extended | |
content | text | | extended | |
Has OIDs: no

jinli=# select relname,reltoastrelid from pg_class where relname=’test_toast2’;
relname | reltoastrelid
————-+—————
test_toast2 | 65544
(1 row)

查询关联的TOAST表

jinli=# select relname from pg_class where oid = 65544;

​ relname

pg_toast_65541
(1 row)

插入数据计算长度

jinli=# insert into test_toast2(id,author,title,content) values (1,’jinli’,’test’,repeat(‘test’,100000));
INSERT 0 1

jinli=# select pg_relation_size(65541),pg_relation_size(65544);
pg_relation_size | pg_relation_size
——————+——————
8192 | 8192
(1 row)

此时已经用到了TOAST，PG_COLUMN_SIZE（4590大于2K）。

jinli=# select pg_column_size(id),pg_column_size(author),pg_column_size(title),pg_column_size(content) from test_toast2;
pg_column_size | pg_column_size | pg_column_size | pg_column_size
—————-+—————-+—————-+—————-
4 | 64 | 5 | 4590
(1 row)
